# 149 — v0.45.10: defensive web-publish recipe

Forensic trace of atlas/strengths (marinade) showed an agent silently
writing files to the right path (`/workspace/web/pub/strengths/`)
then DESTRUCTIVELY moving them to `/workspace/web/atlas/strengths/strengths/`
because `<autocalls>` reported `tier=1, WEB_PREFIX=atlas` and the agent
followed Case B blindly. The `atlas.fab.krons.cx` subdomain doesn't
exist on marinade (`vhosts.json` is `{}`); curl returned 403, agent
ignored its own verification, announced the broken URL, four pages
shipped to anonymous form submitters with non-resolving URLs.

This release rewrites `ant/CLAUDE.md ## How to publish a web page`:

- **Verify-before-announce is mandatory.** Agents may NOT emit a URL
  they haven't curl-verified to return 200. Banner at the top.
- **Fall-through rule.** Case B / Case C now have an explicit
  fall-through: if `curl -sI` returns non-200, do NOT move files,
  do NOT announce, switch to Case A (`/pub/<groupname>/<app>/`,
  served at `host/pub/...`). Single-vhost deployments (marinade)
  trigger this fallback naturally.
- **Anti-patterns named.** Three failure modes from atlas/strengths:
  (a) trusting env vars without verification, (b) moving working
  files to "fix" 4xx, (c) treating 4xx as transient. Each is now
  explicitly forbidden.
- **Operator overlay precedence.** Reiterated: `~/CLAUDE.md`
  overrides the case-by-tier logic. Agent must read it first.

No daemon changes. No schema changes. CLAUDE.md edit + new
migration entry only. Migrate skill 3-way-merges `~/.claude/CLAUDE.md`
across all groups on next migration.
