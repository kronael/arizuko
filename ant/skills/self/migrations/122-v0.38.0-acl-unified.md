# v0.38.0 — unified ACL

The four legacy auth tables (`user_groups`, `user_jids`, `grant_rules`,
the dead `grants`) collapse into two: `acl` + `acl_membership`. Spec at
`specs/6/9-acl-unified.md`. End-state — no legacy paths.

What this means for your skills:

- **One Authorize call site.** Every authorization check now goes
  through `auth.Authorize(s, Caller, action, scope, claims, params)`.
  The structural / tree-shape gates (delegate prefix, tier bound, task
  ownership) stay separate as `auth.AuthorizeStructural`.
- **Membership is one graph.** Role membership, JID claims, role
  hierarchy — all rows in `acl_membership(child, parent)`. Expansion
  is transitive.
- **Operators are no longer magic.** `role:operator` is a regular
  principal with `acl(role:operator, *, **, allow)`. Membership in
  the role is via `acl_membership`.
- **The MCP tool surface changed.** `get_grants` / `set_grants` are
  gone. The new `list_acl(folder)` tool returns acl rows scoped to a
  folder. Operator writes via dashd UI or `arizuko grant`.
- **CLI**: `arizuko grant <sub> <pattern>` now writes `acl(sub, admin,
  pattern, allow)`. `**` pattern → operator role membership.

Spec: `specs/6/9-acl-unified.md`. Operator docs: `/pub/concepts/grants.html`.
