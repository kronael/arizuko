# 150 — v0.45.11: FHS mount rename + ~/public_html and ~/private_html

Two changes:

1. Platform mount paths moved to FHS canonical locations.
2. Each group has two new writable web slots in agent home, served at
   distinct URLs with distinct auth gates.

## The agent web rule

> Your home is `~`. Two web slots, both bind-mounted from the unified
> web tree:
>
> - **`~/public_html/`** → served at `/pub/<your-folder>/...` (no auth)
> - **`~/private_html/`** → served at `/priv/<your-folder>/...` (OAuth)
>
> Off-web storage stays in `~/workspace/`, `~/diary/`, `~/facts/`,
> `~/users/`, `~/.claude/`. Nothing under those is ever served at a URL.
>
> Read-only browse of the whole public web tree at `/var/lib/www/`.

## Platform mount paths

| Old | New | Mode |
|---|---|---|
| `/workspace/self` | `/opt/arizuko` | RO |
| `/workspace/web` | `/var/lib/www` | RO whole pub tree, tier 0-2 |
| `/workspace/ipc` | `/run/ipc` | RW |
| `/workspace/share` | `/var/lib/share` | RO/RW per grant |
| `/workspace/data/groups` | `/var/lib/groups` | RW, tier 0 only |
| `/workspace/extra/<n>` | `/mnt/<n>` | varies |

## How URLs map to files

| URL | Served from | Auth |
|---|---|---|
| `https://<host>/pub/<X>` | `<data>/web/pub/<X>` | none |
| `https://<host>/priv/<X>` | `<data>/web/priv/<X>` | JWT (OAuth) |
| `https://<host>/<X>` | rewrites to `/pub/<X>` (existing) | JWT |

The `/pub/<X>` and `/<X>` URLs serve the SAME file. The `/priv/<X>`
URL serves a DIFFERENT file from a separate filesystem tree. Content
under `web/priv/` is NEVER reachable via `/pub/` URLs.

## Cross-group visibility

`/var/lib/www/` mounts the whole public web tree read-only — you can
see what other groups publish. To link to operator-curated docs:

```html
<link rel="stylesheet" href="/pub/arizuko/assets/hub.css">
```

You cannot write to `/var/lib/www/` directly. Your writable slot is
`~/public_html/` (which IS the same bytes as
`/var/lib/www/<your-folder>/` — bind mounts share inodes).

## Nested subgroups

Tier-2 group `atlas/support` has:
- `~/public_html/` bind-mounted from `<data>/web/pub/atlas/support/`
- Served at `/pub/atlas/support/...`

atlas's parent `~/public_html/` is bind-mounted from
`<data>/web/pub/atlas/` — atlas SEES its subgroup's content via
`ls ~/public_html/support/` (the subdir is a real filesystem path in
the unified tree), but writing under that subdir from atlas's
container affects the same underlying files. **Treat subgroup
directory names as reserved**: check `/var/lib/www/` for existing
content before writing.

## ~/workspace/ stays

Your permanent private (off-web) storage at `~/workspace/` is
unchanged. The platform's `/workspace/*` paths were renamed to FHS
locations, but your agent home's `~/workspace/` subdir keeps its
name.

Skill-managed subdirs (`~/diary/`, `~/facts/`, `~/users/`,
`~/.claude/skills/`) are unchanged.

## Operator extras now under /mnt/

If your group has `additional_mounts` configured in the operator's
group TOML, those now appear at `/mnt/<name>/` instead of
`/workspace/extra/<name>/`. Same content, new path.

Reference: `container/runner.go buildMounts`, `container/runner.go
SetupGroup`, `proxyd/main.go route` (the `/priv/*` handler),
`specs/5/V-web-vhosts.md`.
