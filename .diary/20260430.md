## ~08:00 — transparent mode

User flagged that the original transparent-proxy direction was the
right one and we should ship it. Earlier today (during crackbox v1)
I'd attempted transparent → forward proxy pivot because Docker
bridges put the host on the gateway IP, so iptables REDIRECT in
crackbox's netns saw nothing. The forward proxy worked but loses the
"any client" property — Node fetch needs a shim, anything ignoring
proxy env can't reach the internet.

User's framing: "no opt-ins and bullshit, just enable by default and
can disable in a config like crackboxrc or so." So:

- Crackbox now binds three listeners by default: `:3128` forward
  (HTTP + CONNECT), `:3129` admin, `:3127` transparent (raw TCP +
  `getsockopt(SO_ORIGINAL_DST)` + SNI/Host peek).
- The transparent listener is harmless when nothing redirects to it
  (idle), so binding by default costs nothing.
- Disable via TOML at `~/.crackboxrc` / `$XDG_CONFIG_HOME/crackbox/`
  / `/etc/crackbox.toml`, or `--transparent ""`, or
  `CRACKBOX_TRANSPARENT_ADDR=""`.
- Same daemon, same registry, same matchHost. No "modes" anywhere
  in the daemon — just two ways traffic finds the proxy.

Sub did the implementation. Restored `peek.go` + `origdst_linux.go`
+ `origdst_other.go` from `6439c2b^` (the commit that deleted them
during the earlier pivot). Added `pkg/config/` for TOML loading with
`flag > env > config > defaults` precedence. CLI grew `--config` and
`--transparent` flags. Tests: real TLS ClientHello fixtures, deny
path, unsupported-port, lifecycle, race-clean.

Latent gitignore bug fix on the side: `.gitignore` had unanchored
`HEAD` / `config` / `hooks` / `objects` / `refs` lines (meant for
git-internal nested-checkout files), which silently hid
`crackbox/pkg/config/*` from `git status`. Anchored to leading `/`.

## ~10:00 — production deploy across all three

User flagged that "everybody sees the API failing" — krons users
weren't getting agent replies. Brought all three instances (krons,
sloth, marinade) onto the same v0.31.0+ image with crackbox enabled.

Sequence of fixes during the deploy attempt:
1. **Path-derivation outage.** v0.31.0 deployed via image rebuild
   on krons. gated wouldn't start because EgressCrackbox was empty.
   Root: I'd used `filepath.Base(c.ProjectRoot)` which inside a
   container returns `home` (mount path), not `arizuko_krons`
   (host data dir). User correctly called this strictly prohibited
   wisdom. Recorded in `~/.claude/CLAUDE.md`: never derive identity
   from filesystem paths.
2. **Compose now writes env explicitly.** `EGRESS_NETWORK_PREFIX`
   and `EGRESS_CRACKBOX` derived at compose-generation time (where
   the project name *is* known) and written into gated's env file.
3. **Env-name regression test.** Added TestLoadConfigEgress that
   pins every `EGRESS_*` env var to its config field. Catches the
   class of bug where config reads the wrong env name.
4. **Retry-loop bounded.** `processSenderBatch` advances cursor
   past failed batches instead of resetting. A permanent failure
   like the path-derivation one would otherwise replay every
   inbound message forever.
5. **Subnet-overlap retry.** When `docker network create` says
   "Pool overlaps" (orphan network from prior instance name), the
   allocator retries with the next /24 slot up to 8 attempts.

After all that, sloth and marinade joined krons on the new image
by adding `CRACKBOX_ADMIN_API=http://crackbox:3129` to their .env.

## ~11:25 — "still getting timeouts"

User reported agent replies taking 1-2 min on krons. Investigation:
agent container started, "Session initialized" logged at +1s, then
silence for the next minute. Zero traffic through crackbox in that
window. Container alive but the SDK had nothing to log.

Root cause: DNS for `crackbox` failed inside agent containers on
per-folder networks. Docker compose's service-name alias only applies
to the project's default network. Per-folder networks created at
runtime by gated only resolve container names (`arizuko_crackbox_krons`),
not the short `crackbox` alias. Agent's `HTTPS_PROXY=http://crackbox:3128`
hit DNS-fail → every connect hung indefinitely → user saw nothing for
30 min until the agent's idle timeout fired.

Fix: pass `--alias crackbox` to `docker network connect` in
`container/network.go::connectCrackbox`. After redeploy, traffic
flows: spawn at 11:30:43 → first allow connect at 11:30:47 (~3.6s),
normal agent latency from there.

## ~16:00 — slink round-handle protocol

User wanted a CLI to inject messages into any group from the server
with "SEND + wait until container finishes" semantics. Triaged:

- slink already has the shape — `wait`-on-SSE in `webd/slink.go`,
  capped at 120s, returns first assistant frame.
- That's NOT round-done — first reply ≠ run finished.
- handleSubmitTurn writes to turn_results but doesn't publish to hub,
  so slink can't observe the close anyway.

Designed the round-handle protocol: each agent run = first-class
object keyed by turn_id (= inbound message_id, already the contract
per spec 5/27). Endpoints:

```
POST /slink/<token>                          → {user, turn_id, status:pending}
POST /slink/<token>?steer=<turn_id>          chained injection
GET  /slink/<token>/turn/<id>                snapshot
GET  /slink/<token>/turn/<id>?after=<msg_id> cursor paging
GET  /slink/<token>/turn/<id>/status         cheap status
GET  /slink/<token>/turn/<id>/sse            stream + round_done
```

Almost over-engineered: was about to add `turn_results.inbound_msg_ids`
JSON column. User pushed back: "try to reuse something we already have
... or simplify code possibly elsewhere identifying the turns etc."
Right call. Settled on:

- ONE new column: `messages.turn_id` (migration 0038) + partial index
- Reused `messages.topic` for hub keying (no new hub key — existing
  `folder/topic` handles round-handle SSE; handler filters by turn_id
  in payload)
- No `inbound_msg_ids` because each round-handle injection mints a
  fresh topic, so per-folder queue serializes 1 message per round
  by construction
- `core.Channel.Send` widened by one positional `turnID` (3 impls
  + 6 test sites updated) — cleaner than a parallel TurnAware interface
- Round_done: gateway → webd `POST /v1/round_done` from
  handleSubmitTurn only when the chat is `web:<folder>`. webd
  publishes round_done on existing `(folder, topic)` key.

Steering: SDK doesn't allow mid-turn injection — LLM context is
frozen until the current turn produces its final response. So
`?steer=<turn_id>` reuses the in-flight round's TOPIC so the steer
lands on the same chat queue and becomes the immediate next round.
chained_from set in response. If the round already finished, the
steer becomes a fresh round (no chain).

CLI: `arizuko send <instance> <folder> "<msg>" [--wait | --stream
| --steer <turn_id>]`. ~270 LOC, pure HTTP wrapper. Reads
slink_token + WEB_HOST from instance state, posts via slink, polls
or streams. No new transport, no new auth. Zero server-side code.

Specs touched: `1/W-slink.md` (rewritten with full protocol), plus
consistency edits in `5/27-detached-containers.md`, `5/J-sse.md`,
`5/index.md`, `6/3-chat-ui.md`, `7/3-local-cli.md`, `7/index.md`.

## ~18:30 — duplicate broadcast investigation

User reported they got TWO copies of the v0.32.0 release notice on
krons. Then "rhias seen twice" — appearing to loop. Diagnosed by
walking the messages.db:

- 4 different agents independently broadcast (mayai 18:36, happy
  18:37, rhias 18:41, krons 18:47 — each ~1 min apart)
- Each wrote its own `~/.announced-version` claim file (per-agent
  dedup, no deployment-wide coordination)
- Multiple worlds' agents sent to the same group chat JID
  (e.g., telegram:-5228252682 got 4 copies — one each from mayai,
  happy, rhias, krons)

Initial framing: "fan-out dedup needs gateway-side ledger." User
pushed back: "A can't happen, the bug is elsewhere." Investigation
revealed the true root cause was in two parts:

1. **Gateway broadcast trigger correctly fan-outs `/migrate` to
   every top-level world** (each world IS a peer "root" by design
   per `runner.go:150 root := !strings.Contains(folder, "/")`).
   This part is correct.
2. **`send_message` / outbound MCP tools accept any `jid:` value
   without checking whose subtree it belongs to.** Routing is 1:1
   inbound but unconstrained outbound. Tier-1 worlds were spamming
   cross-world by enumerating JIDs from memory and just calling
   `send`. THIS is the actual bug.

User confirmed: "the worlds should not be able to cross send."

direct-send privilege over the others).
