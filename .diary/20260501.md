## 04:00 — `make images` self-foots image-pull failures

User reported 5 chats had errored messages. Investigation:

- 5 monthly cron tasks fired at 04:00:27 UTC
  (`/compact-memories diary|episodes month` to telegram chats for
  happy, krons, mayai, rhias/content, rhias/nemo)
- All 5 spawns failed with exit 125: `pull access denied for
  arizuko-ant, repository does not exist`
- Root cause: `make images` opens with `docker image prune -af`.
  The `-a` flag deletes ANY image not used by a *running*
  container — including our local-only `arizuko-ant:latest` when
  no agent container happens to be active. Build then takes ~2 min
  to recreate it; cron tasks firing during that window try to spawn,
  Docker tries to pull from a registry, fails (we have no registry).
- This is documented as "Nothing works checklist #1" in CLAUDE.md
  but it shouldn't be self-inflicted by our own release process.

Fix: `e36d941 [fix] make images: drop -a from docker image prune`
— now uses `-f` (dangling only).

Cleanup: cleared `errored=1` on the 5 affected message rows so the
next inbound to those chats doesn't replay the failure. Tasks won't
re-fire until next month (cron `0 4 1 * *`); operator can manually
trigger via timed if desired.

## 04:30 — corrected: web/pub is agent-managed per instance, not synced

User pushed back on the "web/pub template-overlay sync gap" I'd
listed as an open follow-up. They're right — `template/web/pub/`
is the SEED for new instance creation. Once an instance exists,
its `/srv/data/arizuko_<inst>/web/pub/` is the agent's workspace
mount (`/workspace/web/pub/`). The agent owns and edits that dir.
There's no "gap" — it's the design.

When upstream docs improve (e.g., today's `concepts/slink.html`
addition), the agent can pull them from `/workspace/self/template/
web/pub/` (the read-only upstream mount) into its own workspace
when/if it wants. No automated rsync needed. The fact that I manually
`cp -r template/web/pub/slink/ /srv/data/arizuko_krons/web/pub/`
during the v0.32.0 ship was an overreach — I shouldn't have written
to the krons agent's workspace from the host. krons agent can
re-organize its docs as it sees fit. Task #22 deleted.

## ~10:00 — marinade timeout root cause + IDLE_TIMEOUT default fix

User reported "marinade not working". atlas/support container had
been timing out at 1h0m0s repeatedly (4 errored msgs over the day).
Root cause: `IDLE_TIMEOUT=60000` in marinade `.env` parses as 60
SECONDS via `chanlib.EnvDur` (unit is ms, not s). Default is 30 min;
60s killed the container any time the agent went silent during long
think blocks / web fetches. The "Container timed out after 1h0m0s"
error message is misleading — it always reports `cfgTimeout` (the
hard deadline) regardless of which timer fired.

Fix: bumped `core/config.go` IDLE_TIMEOUT default 30min → 60min;
stripped explicit `IDLE_TIMEOUT=` from all 3 instances' `.env` files.
Code change committed (6748614); deploy still pending (`sudo make
image` + restart). Old binary holds old defaults in memory until
restart, so no live regression yet.
