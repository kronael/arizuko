## WEB_PREFIX tier fix (commit 9017e5e)

The rhias/nemo agent burned an hour hunting `/workspace/web/` because
the env claimed a publishable surface (`WEB_PREFIX=pub/rhias/nemo`)
that the mount table denies (tier 2+ has no web mount per spec 4/18).
Two underlying bugs, one fix:

- `container/runner.go:678-693` unconditionally set
  `WEB_PREFIX=pub/<folder>` for every non-root tier. Tier 2+ has no
  mount; tier 1 publishes via vhost subdomain not via `/pub/<world>/`
  path. Now tier-gated:
  - root: `WEB_PREFIX=pub`
  - tier 1: `WEB_PREFIX=<folder>` (vhost subdomain prefix)
  - tier 2+: `WEB_PREFIX=""` (no surface; ask parent)
- `ant/CLAUDE.md` web section replaced the single uniform rule with
  a tier-aware recipe + working code snippet per case + curl
  verification (commit bed88ec). Agent now diagnoses its case from
  `ls /workspace/web/` + `echo $WEB_PREFIX` first.

Deployed: `sudo docker build -t arizuko .` (gated) + `make agent`
(arizuko-ant) + `systemctl restart arizuko_krons`.

## whapd pairing recovery — what to do next time

WhatsApp invalidated the krons session (mass logout from the primary
phone, or account re-login). Symptom: whapd loops `401 session
invalidated, delete auth dir and re-pair`. Backup recovery (tried
all five `whatsapp-auth.bak.*` dirs) was futile — once WhatsApp
revokes a session server-side, NO local backup will re-authenticate.

**Faster pairing: pairing-code mode, not QR.** whapd ships with
`--pair <phone>` (`whapd/src/main.ts:113`), which calls Baileys'
`requestPairingCode`. The user enters an 8-char code on their phone
under "Link with phone number instead" — no camera scan needed.

### Recipe (15 seconds end-to-end)

```bash
# 1. Stop the looping daemon and clear stale auth.
sudo docker stop arizuko_whapd_krons
sudo rm -rf /srv/data/arizuko_krons/store/whatsapp-auth/* \
            /srv/data/arizuko_krons/store/whatsapp-auth/.[!.]*

# 2. Fire a one-shot pair container. NOTE: --log-driver json-file
#    is required (compose sets log-driver=none, blocks docker logs).
sudo docker run -d --name whapd_pair_krons --log-driver json-file \
  -v /srv/data/arizuko_krons/store/whatsapp-auth:/srv/data/store/whatsapp-auth \
  -e WHATSAPP_AUTH_DIR=/srv/data/store/whatsapp-auth \
  -e LISTEN_ADDR=:8080 \
  -e CHANNEL_SECRET=<from .env> \
  -e ROUTER_URL=http://gated:8080 \
  --entrypoint node arizuko-whatsapp:latest \
  dist/main.js --pair +<phone>
sleep 10
sudo docker logs whapd_pair_krons | grep "pairing code"

# 3. Enter on phone: WhatsApp → Settings → Linked Devices →
#    Link a Device → "Link with phone number instead" → enter code.

# 4. After pair succeeds, swap back to normal whapd.
sudo docker rm -f whapd_pair_krons
sudo docker start arizuko_whapd_krons
```

### Production-UX upgrade (TODO, suggested below)

The pairing-code recipe is still operator-only — there's no UI for it
in dashd. Three minimal improvements:

1. **whapd HTTP `/v1/pair/start?phone=+...` endpoint.** Returns
   `{code, expires_at}`; runs the same `pairOnce()` in-process. No
   one-shot container, no docker gymnastics. ~30 LOC patch to
   whapd/src/server.ts.
2. **dashd "WhatsApp re-pair" button.** Calls the endpoint, shows
   the code with a copy button and a countdown. Behind admin auth
   (already wired). ~50 LOC.
3. **Auto-rotate stale auth dir on `401 session invalidated`.**
   When whapd sees three 401s in a row, move the broken auth dir
   aside (`auth.bak.<ts>/`) and enter pair-pending mode automatically
   instead of looping forever. The dashd button surfaces the code.
   Removes the manual "stop + clear + restart" dance entirely.

Spec the trio under `specs/N/<n>-whapd-rebind.md` before shipping.
Total budget ~150 LOC + a dashd page.
