### Format research

Surveyed four best-in-class changelogs in parallel (`WebFetch`):

- **Resend** — 3–6 word headline + 8–15 word body. Date only, no
  version numbers. Most minimal.
- **Linear** — dated sections, multi-feature blocks, fixes/
  improvements grouped by area. Most structured.
- **Vercel** — active voice headlines, code blocks for examples,
  per-release author attribution. Most technical.
- **Cursor** — 150–400 words per release, versioned headers,
  inline subsections.

Common pattern across all four: dated header, terse one-line
headline + bullets, active voice, user benefit before internal
detail, link to canonical source. Differences in length only.

### Converged format

For arizuko's chat broadcast (the existing `>` blockquote convention
from migration 083): Resend-style minimalism with a Cursor-style
versioned header. Spec recorded in root `CLAUDE.md` under new
`## Announcing` section:

- Header: `arizuko vX.Y.Z — DD Mon YYYY`
- 3–6 bullets, lead with biggest user-facing change, active voice,
  user benefit before internal detail (one line each)
- Close with `Full notes: github.com/kronael/arizuko/blob/main/CHANGELOG.md`
- Total ≤ 9 lines visible source
- NO migration numbers / file paths / SHAs — those stay in
  `### Added/Changed/Fixed` (maintainer-facing)
