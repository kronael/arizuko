Pivot: ripped out the custom `arizuko-mcp` Python CLI (shipped in
fa35874 ~10h ago) in favour of apify's `@apify/mcpc`. User asked
"isn't there such tool online?" — yes, and one is literally named
`mcpc`. Researched f/mcptools (general), apify/mcp-cli (HTTPie-style
`key:=value`), steipete/mcporter, FastMCP CLI.

User's sharper point: our `message <jid> <text>` / `file <jid>` shortcuts
were anti-orthogonal — they coupled a general MCP client to specific
arizuko tool names. Like shipping `http post-user` instead of
`http POST /users`. The whole "arizuko-specific" framing was the bug.

apify's mcpc is HTTPie-style orthogonal: `mcpc @s tools-call send_message
jid:="$J" text:="hi"`. Uses a bridge daemon per `@session` (unix socket
+ newline JSON-RPC internally, keeps MCP server stdio alive across CLI
invocations, metadata in `~/.mcpc/sessions.json`). Overkill for our case
(gated.sock is already a long-running server, no stdio spawn to
amortise), but user said "without sessions is fine" — wrapper pattern
in SKILL.md uses per-script short-lived sessions with `trap ... close`.

## MCP CLI landscape (retrospective, retired from NOTES.md)

Tool comparison as of 2026-04-10:

| Tool                 | Stars | Maintainer                                     | Grammar             | Install               |
| -------------------- | ----- | ---------------------------------------------- | ------------------- | --------------------- |
| `f/mcptools`         | ~1.5k | `f` (community, neutral)                       | JSON via `--params` | `go install` / brew   |
| `@apify/mcpc`        | lower | **Apify Inc** (commercial web-scraping vendor) | HTTPie `key:=val`   | `npm install -g`      |
| `steipete/mcporter`  | niche | Peter Steinberger (individual)                 | colon `key:val`     | `npm`                 |
| `jlowin/fastmcp` CLI | high  | FastMCP author                                 | HTTPie              | `pip install fastmcp` |

No Anthropic-official CLI. `claude mcp add` is a config helper, not a
tool-calling client. Decision rationale for `@apify/mcpc`:

1. Best ergonomics for shell scripts (`key:=val` beats JSON-only).
2. OSS neutralises vendor lock-in — forkable, auditable, replaceable.
3. Full MCP surface (`tools-*`, `resources-*`, `prompts-*`, `tasks-*`,
   `logging-set-level`, `ping`, `grep`).
4. Considered alternatives lose on either ergonomics or runtime weight.

Call pattern:

    mcpc connect "socat UNIX-CONNECT:$ARIZUKO_MCP_SOCKET -" @s
    trap 'mcpc @s close' EXIT
    mcpc @s tools-call send_message jid:="$JID" text:="hi"

Non-choice: don't ship a wrapper. The first-pass mistake (commit
`fa35874`, 220-LOC Python with hardcoded `message <jid>` / `file <jid>`
subcommands) coupled a general MCP client to arizuko-specific
vocabulary — like `http post-user` instead of `http POST /users`.
Reframing principle: general protocol tools should be general.
