webdav

concepts / webdav · ← concepts

what it does

Per-group workspaces exposed over WebDAV. Mount https://<host>/dav/<group>/ in any WebDAV-aware client — Finder, Windows Explorer, rclone, cyberduck — to read and write the group’s files: media, logs, diary, facts, users, conversations, skills.

what you see

Open /dav/<group>/ and you get a directory listing of the group’s workspace:

  • media/ — downloaded inbound attachments, by date
  • logs/ — agent run logs (read-only)
  • diary/ — the agent’s dated notes
  • facts/ — durable facts the agent has learned
  • users/ — per-user notes and state
  • conversations/ — chat transcripts
  • .claude/skills/ — the agent’s loaded skills

auth

Same auth as /dash/. Sign in at /auth/login — the JWT cookie that protects the dashboard also gates /dav/. WebDAV clients that don’t carry browser cookies need a Bearer token: get one from /dash/profile and configure your client to send Authorization: Bearer <jwt>.

proxyd checks the user’s X-User-Groups claim against <group> in the path. Without a matching grant the request returns 403.

security

proxyd validates the JWT, strips the Authorization header, then rewrites /dav/<group>/<rest> to /<group>/<rest> and forwards to dufs (which sees only the rewritten path, never the user’s credentials).

common uses

  • browse a group’s files from your file manager
  • sync files in and out with rclone (rclone sync ./local krons:/dav/<group>/media/)
  • mount once and edit memory directly with your usual tools
  • download large attachments without going through the bot

when to disable

Set WEBDAV_ENABLED=false in the instance .env if exposing the workspace over HTTPS isn’t the model you want. The dufs container is omitted from the compose output and proxyd returns 404 for /dav/*.
