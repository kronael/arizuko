webdav

concepts / webdav · ← concepts

Every group’s workspace is a real filesystem you can mount from Finder, rclone, curl, or any WebDAV client at https://<host>/dav/<group>/. For the step-by-step setup see how-to / webdav.

why

Agents read and write files all day — diary, facts, generated drafts, downloaded attachments. You want to drop in a CSV from your laptop, edit a post in your editor, browse what the agent produced — without scp, without pasting blobs into chat, without giving the box shell access.

show me

# list a group's files with curl + PROPFIND
curl -X PROPFIND https://<host>/dav/<group>/ \
     -H 'Depth: 1' \
     -H "Authorization: Bearer $TOKEN"

# or mount it as a real filesystem
rclone mount mygroup:/dav/<group>/ ~/mnt/<group> --vfs-cache-mode writes

Get $TOKEN from /dash/profile — or sign in at /auth/login and the JWT cookie works the same way for browser clients.

what you get

  • JWT/Bearer auth. Same JWT that gates /dash/. No Basic Auth — proxyd’s local OAuth issues Bearer tokens for clients that don’t carry cookies.
  • Per-group scoping. proxyd checks the user’s groups claim against <group> in the path; without a matching grant the request is 403.
  • Sensitive-path write block. .env, *.pem, anything under .git/ can be read but never written via WebDAV.
  • Logs are read-only. Anything under <group>/logs/ answers GET/PROPFIND but rejects PUT/DELETE/MOVE.
  • Browse from anything. Finder, Windows Explorer, cyberduck, rclone, curl — standard WebDAV verbs, no client-specific shim.
  • dufs under the hood. A single Rust binary serves the files; proxyd does auth, path scoping, and the write-block guard before forwarding.
