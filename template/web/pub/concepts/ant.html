ant

concepts / ant · ← concepts

An ant is a Claude agent defined by a folder. Drop in SOUL.md for persona, skills/ for tools, secrets/ for credentials — and arizuko (or ant standalone) runs it.

why

Chat agents need persona, skills, secrets, and a memory that survives restarts. Most apps reinvent that layout from scratch — an env var here, a config file there, a prompt template in a Python string. Ant is the layout: one folder, one shape, forkable and version-controllable. Copy the folder, tweak SOUL.md, ship a new agent.

the folder is the agent

my-agent/
  SOUL.md          — persona, voice, identity
  CLAUDE.md        — operator runbook the agent reads each turn
  skills/          — SKILL.md files (custom + curated)
  diary/           — running log, persistent across runs
  secrets/         — folder-scoped credentials, env-injected
  MCP.json         — optional, declares MCP servers wired in
  workspace/       — the agent’s working directory

That’s the unit. Everything else — CLI, sandbox, orchestrator — is delivery mechanism.

three commands

One binary, three IO modes by flag, sandbox orthogonal:

ant <folder>                              # interactive chat (stdin/stdout)
ant <folder> --prompt="<text>"            # one-shot, exit on completion
ant <folder> --mcp [--socket=<path>]      # expose as an MCP server
ant <folder> --sandbox=none|dockbox|crackbox

The runtime is the same in every mode: spawn claude with the folder’s skills, secrets, and workspace mounted; route diary writes to <folder>/diary/; wire MCP servers per MCP.json. Flags only pick where input comes from and where output goes.

what you get

  • Persona that survives restarts. SOUL.md is read every turn — identity is in the folder, not in a session.
  • Curated skills. skills/SKILL.md entries pick the tools the agent gets. No plugin system; what’s in the folder at spawn time is what runs.
  • Folder-scoped secrets. Per-agent credentials in secrets/ instead of one big env-soup shared across every agent on the host.
  • Persistent diary and memory. Files written to diary/ and facts/ (resolved to ~/diary/ and ~/facts/ inside the container; see ant/CLAUDE.md Storage table) stay across runs. Restart, fork, copy — the agent remembers because the folder remembers.
  • Optional sandbox. --sandbox=dockbox for Docker isolation; --sandbox=crackbox (when crackbox ships its host backend) for tighter egress control.

memory: compaction on a different cadence

Most chat-agent systems run a context reducer — a cheap-model summariser that condenses the running message history before each turn (e.g. muaddib’s src/rooms/command/context-reducer.ts). arizuko inverts the cadence. In-session context stays raw; Claude Code’s own compaction bounds it. Compression is a separate, longer-horizon job:

  • Cron-driven, day/week/month — the /compact-memories skill rolls JSONL session transcripts into episodes/YYYYMMDD.md, then weekly, then monthly. Diary entries are folded the same way.
  • Agent-invoked on demand — /distill compresses one specific input on request, writing durable content (skill, fact, episode) rather than an ephemeral prompt block.
  • Recall, not re-summarise — /recall-memories pulls relevant slices back when needed; the prompt prepends a small <knowledge layer="diary"> block at every turn.

Same problem — token budget vs. fidelity — at a different point on the axis. No silent per-turn condenser; compaction is durable and inspectable on disk.

arizuko vs standalone ant

Today: every arizuko group is an ant folder. Each groups/<folder>/ already has the shape above — SOUL.md, CLAUDE.md, skills/, diary/, secrets/. arizuko chat <instance> [group] drops you into one. gated spawns containers from the same folders for inbound messages.

Planned: a standalone ant <folder> CLI that runs one agent without arizuko anywhere on the system — no gateway, no scheduler, no router. Spec at specs/5/b-ant-standalone.md; status: unshipped, packaging in flight.
