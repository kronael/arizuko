tasks

Skills decide what an agent can do when prompted. Tasks decide when a prompt arrives — including when no human is around to send one. A task is a saved prompt with a schedule. When the schedule says it’s time, the prompt lands in a folder as if a user had just typed it, and the agent answers like any other turn. That last part is the trick worth keeping: a scheduled run is not a special code path, just a synthetic message on the same rails everything else rides.

what it is

One row in the scheduled_tasks table = one recurring or one-shot agent run. The body is a prompt string; the trigger is either a cron expression, an interval in milliseconds, or a single future timestamp.

mechanism

The timed daemon ticks every 60 seconds. On each tick it claims every row where status='active' and next_run ≤ now, flips them to 'firing', then for each claimed row inserts the prompt into the messages table with sender timed (or timed-isolated:<id>) and the task’s chat_jid.

routd polls messages the same way it polls every other inbound source. It sees the new row, routes by chat_jid, and fires a turn. The agent has no idea this prompt came from a clock instead of a person — same code path.

After insert, timed writes the next next_run (parsed from the cron expression or added from the interval), or marks the row 'completed' if there is no schedule. Every fire writes one row to task_run_logs with duration and status.

triggers

The cron column carries one of three shapes:

  • Cron expression — standard 5-field cron (0 9 * * 1-5 for weekday mornings). Parsed by robfig/cron in the task’s timezone (the daemon’s TZ env, default UTC).
  • Interval — a plain integer string of milliseconds. 1800000 = every 30 minutes. The next next_run is just now + interval after each fire.
  • One-shot — empty cron. The row fires once at next_run, then flips to status='completed' and stops being polled. The row stays in the table for audit; nothing deletes it.

context mode

One column on the row decides what session the agent runs under. With group (the default) the synthetic message hits the folder’s main session like any other turn — the task sees the folder’s prior chat history and shares state with it. With isolated the sender becomes timed-isolated:<task-id>, which routd treats as its own conversation — handy for a recurring health check that shouldn’t pollute the human chat’s context. The full scheduled_tasks column list is in reference / schema.

managing tasks

Agents whose tier permits scheduling get MCP tools to create (schedule_task, which dedupes on prompt + chat + schedule so a repeated call returns the existing ID), list, pause, resume, and cancel tasks, plus inspect_tasks — which joins task_run_logs to show when each last fired and whether it succeeded. Full signatures: reference / mcp. There is no arizuko task CLI subcommand today; operators reach the rows through the agent (arizuko chat) or sqlite3 directly.

failure model

If the synthetic message insert itself fails, timed logs an error row to task_run_logs and flips the task back to active without advancing next_run. The next tick will try again.

If the insert succeeds but the agent run downstream errors out, timed never knows — from the scheduler’s point of view the fire was successful. next_run advances normally and the task fires again at its scheduled time. There is no automatic retry of the failed turn. The error sits in routd’s logs and the folder’s chat history; an operator (or the agent itself, on its next run) decides what to do.

The status field never auto-flips to paused based on failures. A failing recurring task keeps firing until somebody pauses it or fixes the underlying problem.
