slink

concepts / slink · ← concepts

One public URL per group. Open it in a browser for a chat widget; POST JSON to it from a script and the agent runs — one container spawn, one reply, paged by polling, by SSE, or via MCP.

why

An agent is only useful if you can reach it from where the work happens: a web page, a shell script, a CI job, an MCP-capable client. Slink is the lowest common denominator: a token in a URL, a POST with text, an HTTP response carrying a handle for the round that just started. Same surface for the browser chat widget, curl from a cron job, agent-to-agent calls, and Claude Desktop over MCP.

show me

# post a message, get back a turn_id (the round handle)
curl -X POST https://<host>/slink/<token> \
     -H 'Accept: application/json' \
     -d 'content=summarize today's diary'
# {"user":{"id":"msg_abc",...},"turn_id":"msg_abc","status":"pending"}

# stream the reply over SSE; closes on round_done
curl -N https://<host>/slink/<token>/msg_abc/sse

# or: poll for new frames since the last one you saw
curl https://<host>/slink/<token>/msg_abc?after=out_002

what you get

  • Token-based auth. 32 random bytes in the URL. Public by design — share it, bookmark it, embed it in a script.
  • Rate-limited at the edge. proxyd caps /slink/* per client IP so a leaked token is a leaked rate-limit bucket, not a leaked password.
  • Browser chat at GET /slink/<token>. Drop-in widget — no signup, no SDK. Same token your scripts use.
  • Round handle for machines. POST returns a turn_id; GET /<id> snapshots, ?after=<last> pages, /sse streams until round_done.
  • Steering. POST /slink/<token>/<turn_id> chains a follow-up onto an in-flight round, or starts a fresh one if the round already finished.
  • MCP transport on the same path. POST /slink/<token>/mcp exposes three tools — send_message, steer, get_round — for Claude Desktop or any MCP client.
