what it is

A JID identifies one resource on one platform. It’s the wire-form key arizuko uses for chats, senders, and routing rules. Shape:

<platform>:<rest>

Each platform declares its own <rest> shape with a kind in the first segment, separated from the rest by /.

routing globs

Routes match a JID with path.Match glob semantics — * matches any non-/ sequence (segments are first-class). So:

  • chat_jid=telegram:group/* — all telegram groups
  • chat_jid=discord:67890/* — guild 67890, any channel
  • chat_jid=discord:dm/* — all Discord DMs
  • chat_jid=whatsapp:*@g.us — all whatsapp groups
  • sender=telegram:user/[0-9]* — any numeric telegram user

Bracket character classes […] work; ? matches one non-/ char.

why kinds

Without a kind, multiple resource types collide on the same prefix. Telegram chat IDs are signed integers — positive=DM, negative=group — so old rules had to encode the sign-bit hack. Reddit thing IDs share the reddit: prefix between t1_ (comment), t2_ (user), t3_ (submission). Discord stored just the channel and lost the guild.

One JID = one resource. The kind discriminator goes in the wire form, so routing globs scope by kind without per-platform tricks.

typed in code

Go types: core.JID wraps *url.URL; core.ChatJID and core.UserJID embed it and validate the kind segment on parse. The wire format is a string — database/sql Scanner/Valuer and encoding/json marshalers preserve it everywhere. core.MatchJID(pattern, value) applies the routing glob.

migration

Migration 0038-typed-jids.sql rewrites every JID-shaped value in the store: messages, chats, user_jids, grants, onboarding, and the predicates inside routes.match. Idempotent — safe to re-run. Discord rows without a stored guild_id migrate to the placeholder discord:_/<channel>; new inbound carries the real guild going forward.
