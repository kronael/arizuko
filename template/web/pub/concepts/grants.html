authorization

One question, one row: may principal P perform action A on scope S? Two tables — acl for permissions and acl_membership for identity indirection — cover humans, agents, channels, and roles. One Authorize() call answers every check.

the mental model

arizuko has principals, actions, and scopes. A principal is anyone or anything asking — an OAuth-bound human (google:114alice), an agent container (folder:atlas/eng), a channel room (discord:837…/1504…), or a named role (role:operator). An action is what they want to do — interact, admin, mcp:<tool>, or *. A scope is the folder path they want to act on, globbed segment-wise. Permissions live in one table (acl); identity-and-role indirection lives in another (acl_membership). That's the whole system.

three common patterns

1. user-bot — alice has her own folder

One human, one folder, one grant. No membership row needed.

INSERT INTO routes (match, target) VALUES
  ('platform=telegram room=user/alice', 'alice');

INSERT INTO acl (principal, action, scope, granted_at) VALUES
  ('google:114alice', 'interact', 'alice', now());

Operator CLI:

arizuko grants add google:114alice interact alice

2. channel-bot — shared discord room

One route, one acl row keyed on the room, zero per-user rows. Everyone arriving via the route picks up interact.

INSERT INTO routes (match, target) VALUES
  ('platform=discord room=837.../1504...', 'main/lab');

INSERT INTO acl (principal, action, scope, granted_at) VALUES
  ('discord:837.../channel/1504...', 'interact', 'main/lab', now());
arizuko grants add discord:837.../channel/1504... interact main/lab

3. hybrid — channel + one elevated admin

The channel-bot rows above, plus one acl_membership edge and one role grant. Adding more admins later is one more membership row.

INSERT INTO acl_membership (child, parent, added_at) VALUES
  ('discord:user/811...', 'role:lab-admin', now());

INSERT INTO acl (principal, action, scope, granted_at) VALUES
  ('role:lab-admin', 'admin', 'main/lab', now());
arizuko roles add lab-admin admin main/lab
arizuko role-members add lab-admin discord:user/811...

identity claiming via OAuth

A user posts in Discord as discord:user/811…. Later they log in via Google. The OAuth callback writes:

INSERT INTO acl_membership (child, parent, added_at) VALUES
  ('discord:user/811...', 'google:114alice', now());

That edge is the claim. Next message from the Discord JID expands to include the canonical sub; any grants on google:114alice apply transitively. No row rewriting, no migration — the channel-side identity keeps working unchanged.

roles — Postgres / IAM shaped

One member edge, one or more permission rows on the role. Two operations: add the role to the person, edit the role's permissions.

-- Define the role: editors may admin the docs subtree.
INSERT INTO acl (principal, action, scope, granted_at) VALUES
  ('role:editor', 'admin', 'docs/**', now());

-- Bind alice to the role.
INSERT INTO acl_membership (child, parent, added_at) VALUES
  ('google:114alice', 'role:editor', now());

Adding another editor — one row in acl_membership. Granting editors a new scope — one row in acl. Role-of-role works the same way: acl_membership(role:senior-editor, role:editor).

denies and overrides

The effect column is allow (default) or deny. Deny wins: any matching deny row rejects the request, even if matching allow rows exist.

Ban one user from a channel they would otherwise enter via the route:

INSERT INTO acl (principal, action, scope, effect, granted_at) VALUES
  ('discord:user/badguy', 'interact', 'main/lab', 'deny', now());

Revoke a tier default for a single folder (subtractive override):

INSERT INTO acl (principal, action, scope, effect, granted_at) VALUES
  ('folder:atlas/private', 'mcp:send_file', 'atlas/private', 'deny', now());

For the common case of removing access, prefer deleting the membership or allow row over inserting a deny — see anti-patterns.

the agent's perspective

An agent at folder atlas/eng is the principal folder:atlas/eng. Its grants come from two sources: tier defaults derived in code from path depth (min(slashes, 3)) plus the routed-platform set, and operator overrides as acl rows with action='mcp:<tool>'. Tier defaults are not materialised; they live in grants.DeriveRules for performance and clarity.

When the agent invokes a tool, the in-container MCP server calls:

Authorize(
  principal: "folder:atlas/eng",
  action:    "mcp:send",
  scope:     "atlas/eng",
  claims:    {tier: "2", world: "atlas"},
  params:    {jid: "telegram:group/-1234"},
)

The check tries explicit rows first, then falls back to tier defaults for mcp:* actions. Denied calls return a structured error the agent surfaces — no pre-announced grant list, no “agent asks before trying.”

anti-patterns

  • Don't insert deny rows to revoke access. Remove the membership edge or the allow row instead. Denies are for true exceptions (banning one user from an otherwise-open channel), not for “forget I said that.”
  • Don't grant admin with a params predicate. Conditional admin (admin(jid=…)) is surprising and brittle — admin is by definition full write access at scope. Use mcp:<tool> with params for narrow capabilities.
  • Don't encode routing in ACL. The routes table answers “where does this message land”; acl answers “may this principal act here.” They share the principal namespace for room JIDs but never merge. Outbound subtree containment (IsAuthorizedRoutingTarget) stays structural.
  • Don't pre-list grants in the agent's manifest. Agents discover authorization at the failure site — the gateway returns a denied error; the agent surfaces it. Mid-conversation revoke takes effect on the next message.
  • Don't bypass roles for the operator group. The first operator is bound via one acl_membership(sub, role:operator) edge at arizuko create. Adding operators later is another membership row — not a fresh wildcard acl row each time.
