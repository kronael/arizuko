grants

what an agent can do · ← concepts

What an agent can do is computed at spawn time from three layers: tier defaults from path depth, per-folder rule overlay, and user×folder ACL — composed by the auth package. The agent never asks; the rules are pre-baked into its MCP tool list.

why

A multitenant agent system needs scoped permissions, but the agent shouldn’t have to enforce them or even know the full ACL. Asking a model “is this allowed?” on every call is slow and unreliable; shipping the whole permission table into the prompt is expensive and leaks structure. So we compute the agent’s effective rule set at spawn and bake it into the MCP tool list. The agent literally cannot see tools it isn’t allowed to call, and each call is re-checked at the IPC boundary. Authorization is the daemon’s problem, not the model’s.

the three layers

  • groups — folder hierarchy. The folder path is the identity; tier is derived from depth (min(slashes, 3)). atlas is tier 1 (a world), atlas/support is tier 2, atlas/support/oncall is tier 3.
  • user_groups — (user_sub, folder, granted_at) rows; the folder column is a glob. A row (github:alice, atlas/**) lets alice act on atlas and any descendant. ** = operator. No matching row = no access.
  • grant_rules — per-folder rule overlay table. Strings appended to the tier defaults at spawn time. Managed via the get_grants / set_grants MCP tools (tier 0–1 only) or the operator dashboard.

tiers

  • Tier 0 — instance root. Unrestricted (["*"]). CLI-only surface.
  • Tier 1 — world (top-level folder). Basic send + per-platform send/post/like/… for every platform routed anywhere under the world + management tools (register_group, escalate_group, delegate_group, route management, task management) + read-write share mount.
  • Tier 2 — depth-2 group. Basic send + per-platform actions, but scoped only to platforms routed at this folder or its descendants. Read-only share mount.
  • Tier 3+ — deep groups. Reply-only: reply, send_file, like, edit.

show me

Rule grammar: [!]action[(param=glob,…)]. Last match wins. No match = deny.

send(jid=telegram:group/*)              # can send to any telegram group
!post                                   # cannot create new top-level posts
send_file(jid=*)                        # can attach files to any allowed target
delete                                  # can delete own messages
share_mount(readonly=false)             # workspace mount writable

Spawn-time composition for folder atlas/support (tier 2) with one telegram route under atlas/:

send, send_file, reply
send(jid=telegram:*), post(jid=telegram:*), like(jid=telegram:*), …
share_mount(readonly=true)
<anything in grant_rules for atlas/support>

what you get

  • Tier-derived defaults. No per-folder config needed for the common case — depth picks the rule set.
  • Per-folder overlay. grant_rules table when you need to widen or narrow a specific folder; appended after the tier defaults so a later !action wins.
  • User×folder ACL with globs. ** for operators, atlas/** for world owners, atlas/support for direct grants. Match logic in auth.MatchGroups; segment-wise, * never crosses /.
  • JID–folder containment on outbound. Even if the rule list allows send(jid=telegram:*), auth.Authorize resolves the target chat’s owning folder via routes and refuses anything outside the agent’s subtree. A tier-1 world agent can’t post into another world’s chats.
  • Rules baked into the MCP tool list. The in-container MCP server filters tool registration by these rules and re-checks each call — the agent literally can’t see disallowed tools, and a synthesized call still gets refused at the boundary.

grants as the tool pre-filter

Other agent systems narrow the advertised tool set with a tool pre-filter — a cheap-model classifier picks ≤K relevant tools per turn before the main model sees the list (e.g. AnythingLLM’s “Intelligent Skill Selection”). arizuko reaches the same intent — don’t waste model context on tools the caller can’t use — via grants, evaluated at a different point: spawn-time, once per container run, against the user_groups ACL plus capability rules. The composed rule list is the filter; grants.MatchingRules drops unauthorised tools at MCP registration so the agent never sees them in tools/list, and grants.CheckAction re-checks each call. Static pre-filtering misses “this turn doesn’t need bluesky tools even though the folder has them”; dynamic pre-filtering pays a classifier round-trip and risks hiding a tool the agent wanted. arizuko’s bet: an agent’s effective tool set is a property of the folder, not the sentence.
