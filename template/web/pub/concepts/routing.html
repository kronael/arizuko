routing

how a JID lands in a folder · ← concepts

A small table of jid → folder rules turns inbound platform messages into agent runs in a specific group. Add a row, the next message lands.

why

Inbound messages from every adapter — Telegram, Discord, web slinks, Mastodon, email — arrive in the same store and need to find the right ant folder to run in. Hardcoding a chat-to-folder map in code breaks the moment a second instance wants different layout; per-adapter plumbing breaks differently in every adapter. The routes table is the one indirection: predicates over the message, evaluated in seq order, first match wins.

show me

seq    match                                  target
---    -----                                  ------
-10    chat_jid=telegram:user/12345           atlas/legal
0      platform=telegram                      atlas/content
0      platform=discord room=dm/*             atlas/dm
0      platform=reddit verb=post              atlas/posts
0      chat_jid=web:slink/abc123              solo/chat
0      platform=discord                       atlas/{sender}
9999   (empty)                                atlas              # catchall

Predicates are space-separated key=glob, AND’d. Glob is path.Match: * matches a non-/ run, ? one char, […] a class. {sender} in the target expands to a sanitized sender ID for per-user child groups.

what you get

  • Five match keys. platform, room, chat_jid, sender, verb. Omit a key to leave it unconstrained; key= requires it absent; key=* requires it present.
  • Glob on every field. Same path.Match semantics across all keys — chat_jid=telegram:group/*, sender=telegram:user/[0-9]*, room=dm/*.
  • Reply chain takes precedence. When a user replies to a bot message, the gateway looks up routed_to on the original and sends the reply back to the same group, regardless of what the routes table says.
  • Sticky routing per chat. @atlas/social as the entire message body locks the chat to that group; #support locks the topic; bare @ or # clears. Stored on the chats row, shared by everyone in the chat.
  • Per-user fan-out. {sender} in the target gives each sender their own child group — atlas/dc-alice, atlas/dc-bob — without one route per user.
  • Edit live. The agent has add_route, delete_route, get_routes, set_routes as MCP tools; the operator dash (dashd) shows the table; changes take effect on the next inbound message, no restart.
