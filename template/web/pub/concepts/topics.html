topics

transient work-units inside a group · ← concepts

A topic is one conversation, overlaid on a group — not a path level. Groups persist; topics complete. Many topics can live in the same group at once, and each carries its own Claude Code session so context doesn’t bleed between threads.

what a topic is

Every message row in the store carries a topic column. A topic is just a string — #tickets-42, a Discord thread channel ID, or empty for the group’s default conversation. The (folder, topic) pair keys the sessions table, so a fresh topic gets a fresh Claude session and a fresh context window. Same folder, same skills, same memory, same secrets — just a different conversation pointer.

This keeps the group hierarchy doing one job (where the ant lives, what it can do) and topics doing another (which conversation it is in right now). A support folder taking thirty concurrent tickets is still one folder; it’s thirty topics.

lifecycle

Topics get created four ways:

  • Inline prefix. A user message starting with #name opens topic #name in the current group, persists it on the chat, and the rest of the message is delivered into that topic.
  • /new #name. Drops any existing Claude session for (folder, #name), then injects the remainder of the message as the first turn of a clean topic.
  • Discord threads. A message in a public or private thread automatically carries the thread’s channel ID as its topic — one Discord thread = one arizuko topic, with no operator action.
  • Telegram forum topics. When a message belongs to a Telegram supergroup forum topic, the MessageThreadID rides through as the arizuko topic.

Topics end the same shapes they start: /new on its own clears the default-topic session; /new #name clears a named one; the agent can call the reset_session MCP tool to drop its own session when context drifts. The group itself never goes anywhere — only the conversation pointer resets.

the active topic

Each chat row carries an optional sticky_topic. At any moment a chat has zero or one active topic; inbound messages without an explicit topic inherit that sticky, and the agent’s session is keyed accordingly.

user> #q3-launch what’s the latest on shipping?
bot>  topic → q3-launch
bot>  <reply scoped to a fresh q3-launch session>

user>  any blockers?
bot>  <continues the q3-launch conversation—sticky_topic carried it>

user> #
bot>  topic reset to default
user>  back to small talk
bot>  <replies in the default-topic session>

A bare # clears the sticky topic. After clearing, new messages land in the default (empty-string) topic until something explicitly opens another one.

topics vs groups

It’s tempting to read rhias/support/#tickets-42 as a deeper path. It isn’t. rhias/support is the folder; #tickets-42 is the conversation. One container, one set of skills, one memory tree — the topic just decides which Claude session resumes when the next message arrives.

If two threads need different skills, different memory, or different secrets, that’s a folder split, not a topic split. Topics are cheap (just a string on a message); folders are heavy (a container, a workspace, an MCP socket, an ACL scope).

topics and observed messages

Every trigger turn on a folder also surfaces messages from other chats routed to the same folder, as observed messages — cross-cutting context the agent didn’t reply to directly. That trailing window is folder-scoped: it doesn’t filter by topic. A turn firing in topic #tickets-42 sees observed messages from every chat into the folder regardless of which topics they belong to.

This is the design. Folders are the awareness boundary; topics are the conversation focus. If you need strict topic isolation — no cross-talk, no shared observation window — split into separate folders. Operators who use topics for organisation inside one folder accept the folder-wide observation as the price.

when to use them

Most channels don’t need explicit topics. A 1:1 chat is the conversation; the default empty topic works forever. Topics earn their keep when one group hosts multiple parallel threads that shouldn’t share a context window:

  • A support folder handling many independent tickets in parallel.
  • A research folder running concurrent investigations — each topic preserves its own line of thought.
  • A Discord guild where threads should each get their own session (handled automatically).

Don’t reach for topics to organise files or skills — that’s what folders are for. Reach for them when you want context isolation without losing the shared workspace.
