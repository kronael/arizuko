auth

login + account linking · ← concepts

Log in with any provider — GitHub, Google, Discord, Telegram, local username/password — and link as many as you want to one canonical user. Single resolve at JWT mint; no ghost accounts.

why

People accumulate identities. GitHub at work, Google at home, a Telegram account on the phone, a local username carved out for ops. They’re all the same human, and any one of them should land on the same arizuko user with the same grants. Auto-creating a fresh account on every new OAuth provider is the easy default and the wrong one — it splinters one human into N ghost accounts that share no state. Refusing the second login is also wrong. The right move is to ask once, link explicitly, and from then on resolve every sub to its canonical owner at the only place that matters: JWT mint.

providers

  • GitHub — OAuth2, sub prefix github:; optional GITHUB_ALLOWED_ORG members-only check.
  • Google — OAuth2 + OIDC, PKCE, sub prefix google:; optional GOOGLE_ALLOWED_EMAILS glob allowlist (only trusted when Google asserts email_verified).
  • Discord — OAuth2 + PKCE, sub prefix discord:.
  • Telegram — Login Widget, HMAC-verified against SHA256(bot_token) with a 5-minute auth_date window, sub prefix telegram:.
  • Local — username + argon2id hash, sub prefix local:; rate-limited to 5 attempts per 15 min per IP.

the JWT path

Two tokens. The access JWT is 1h-TTL, lives in localStorage, carries {sub, name, provider, exp}, and rides as Authorization: Bearer on API calls. The refresh token is 32 random bytes, stored in the DB as a SHA-256 hash, returned to the browser as an HttpOnly; SameSite=Strict; Secure; Path=/auth cookie with 30d TTL. Each refresh rotates the cookie single-use — one stolen refresh token is good for exactly one swap before it’s invalidated.

linking accounts

  • Log in with any provider — that sub becomes your canonical identity.
  • Visit /dash/profile — the page lists every linked sub and shows a “Link…” button per supported provider.
  • Click one — the browser hits /auth/{provider}?intent=link&return=/dash/profile/. The redirect handler reads your existing session, encodes intent=link + your canonical sub into the signed OAuth state cookie, and sends you to the provider as usual.
  • The callback writes auth_users.linked_to_sub = <your canonical> for the new sub and refreshes your session unchanged. The provider you were already logged in as never blinks.

collision

Logged in as A, you OAuth as B without an explicit link intent — or with link intent, but B is already canonical for someone else. The callback doesn’t pick a winner: it renders a collision page with two buttons and a 10-min HMAC-signed token.

  • Link to current. Commit linked_to_sub = A, refresh the session as A. Disabled when B is canonical for an existing other user (merging two real accounts is a manual operator step).
  • Log out, become B. Drop the refresh cookie + session, log in as B (creating a canonical row if new).

No silent ghost-account creation, no ambiguous merges.

resolve point

auth.issueSession calls store.CanonicalSub(sub) exactly once before minting the JWT. The minted access token, the refresh row in the DB, and every X-User-Sub header proxyd stamps downstream all carry the canonical sub. gateway, webd, ipc, and the rest never re-resolve — they treat the sub they see as authoritative.

what you get

  • Five providers + local password. Pick whichever is convenient; the system doesn’t care which one is “primary.”
  • One canonical user, many subs. auth_users.linked_to_sub points each linked row at the canonical row. LinkSubToCanonical rejects chains — depth is always one hop.
  • Collision UX, never silent dupes. Seven-case dispatcher, two-button HTML page, signed token. The user always picks.
  • Single resolve point. One CanonicalSub call at JWT mint. Downstream code is sub-shaped data, not a graph traversal.
  • 1h access + 30d refresh, single-use rotation. Bearer header for API, HttpOnly cookie scoped to /auth for everything else.
  • Stateless OAuth state. HMAC-signed cookie, 10-min TTL, optional base64url payload carrying intent=link + canonical sub. No server-side state table.
