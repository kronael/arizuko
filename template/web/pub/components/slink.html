slink

What it is

A slink is a public URL that opens a chat with one group’s agent. The URL looks like https://<host>/slink/<token>. There is no login. Whoever holds the URL can talk to that group.

The <token> is 32 random bytes, base64url-encoded (~43 characters). It is stored in the column groups.slink_token. Every request to /slink/<token>/* looks the token up and resolves it to one folder.

Why it exists

Most arizuko channels (Telegram, Discord, WhatsApp) tie a chat to a platform account. Sometimes a group needs a chat that anyone can open from a link — a support page, a public demo, a script that talks to the agent. Slink is that link.

One group can have one slink token, minted when the group is created. It is not rotated and does not expire. Treat the URL like a public chat-room invite, not a password — the defence is rate limiting and the group’s grants, not keeping the token secret.

The URL shape

https://<host>/slink/<token>            → chat page (HTML)
https://<host>/slink/<token>        POST → send a message
https://<host>/slink/<token>/<id>/sse  → stream the agent’s reply
https://<host>/slink/<token>/mcp   POST → talk over MCP

Open the bare URL in a browser and you get a small chat page. POST JSON to the same URL and you get a turn_id back — the handle to the round the agent is now running.

How it fits

browser / curl / MCP client
        |
        v   /slink/<token>/...
        |
      proxyd   public route, strips identity headers, runs the DoS shield
        |
      webd     resolves token, drops message, serves SSE/snapshot
        |
      gated    runs the round in the group’s container, writes frames to messages.db

proxyd handles /slink/* as a public path: no JWT needed, identity headers are stripped, and anonymous callers go through a rate shield (10 requests per minute per IP by default, controlled by SLINK_ANON_DOS_RPM). Excess returns HTTP 429.

Real-time without WebSockets

The transport is plain HTTP. The browser POSTs a message; the server replies with a turn_id. The browser then opens an SSE stream at /slink/<token>/<turn_id>/sse and reads event: message lines as the agent produces them. The stream closes on event: round_done.

If the connection drops, the browser reconnects with a Last-Event-Id header. The server replays the frames the client missed, then resumes the live stream.

Quick try

arizuko create demo
arizuko run demo
# the default group’s token is in groups.slink_token

# send a message
curl -X POST https://<host>/slink/<token> \
     -H 'Content-Type: application/json' \
     -d '{"content":"hello"}'
# {"user":{"id":"msg_abc",...},"turn_id":"msg_abc","status":"pending"}

# stream the reply
curl -N https://<host>/slink/<token>/msg_abc/sse

Three faces, one path

Built-in chat page — GET /slink/<token>

A minimal HTML chat UI shipped inside the webd binary. Share the URL; the recipient gets a working chat.

JS SDK — /assets/arizuko-client.js

A vanilla JS client served by webd with permissive CORS. A third-party page can embed it and call Arizuko.connect(token) to drive the protocol.

MCP — POST /slink/<token>/mcp

A Streamable HTTP MCP endpoint with three tools: send_message, get_round, get_round_status. Another agent can register the URL and talk to the group as if it were a local tool.

What slink does not do

Slink is a chat surface for users. It does not expose admin actions, the operator dashboard, the WebDAV workspace, or per-user mounts. Those live behind proxyd’s authenticated routes. Slink is one folder, one chat, no login.
