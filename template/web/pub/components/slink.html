slink

What is it

A token-gated HTTP surface that lets any client — browser, curl, or an MCP-speaking agent — submit a message into a group and observe the resulting round.

Why does it exist

One group has many possible consumers: the built-in browser chat page, a third-party page using the JS SDK, a shell script polling for an answer, an external agent calling MCP tools. Without slink each consumer would re-implement the round-handle dance — POST a message, get back a turn_id, stream or poll until round_done, optionally steer with a follow-up. Slink is that dance, exposed once, behind a public token. One renderer, many sinks: every consumer hits the same /slink/<token>/* paths so the contract can't drift between them.

How does it fit

browser / SDK / MCP client / curl
        |
        v   POST /slink/<token>[/<turn_id>]
        v   GET  /slink/<token>/<turn_id>[/sse | /status]
        v   POST /slink/<token>/mcp
        |
      webd     route + token resolve + rate limit + dispatch
        |
        v   inbound message + turn_results row
        |
      gated    durable round in messages.db, spawns container
        |
        v   submit_turn -> POST /v1/round_done -> SSE round_done -> stream closes

Token lives in groups.slink_token (32 random bytes, base64url, never rotated). webd owns /slink/*; rate limits per anon-shared-bucket / per-JWT-sub / unlimited for operator. proxyd already covers /slink/* as a public route — identity headers are stripped on inbound so the token is the only auth surface.

Standalone usage

slink is a route on webd, not a separate binary. Running it standalone means running webd + gated (which owns the schema and the round lifecycle) + at least one group so a slink_token exists.

arizuko create demo
arizuko run demo
# default group is created on first run; the token is in groups.slink_token

# post a message, get the round handle
curl -X POST https://<host>/slink/<token> \
     -H 'Content-Type: application/json' \
     -d '{"content":"hello"}'
# {"user":{"id":"msg_abc",...},"turn_id":"msg_abc","status":"pending"}

# stream the round; server closes on round_done
curl -N https://<host>/slink/<token>/msg_abc/sse

Full endpoint table, payload shapes, SSE event grammar, and steering semantics: reference/slink.

Three faces

Built-in chat page — GET /slink/<token>/chat

The reference HTML chat UI shipped with webd. GET /slink/<token> redirects here. No setup — share the URL.

JS SDK — /assets/arizuko-client.js

Vanilla-JS client served from webd's embedded asset FS, CORS-open. Arizuko.connect(token) returns a handle with send, steer, status, snapshot, stream. Used by branded chat pages on third-party origins.

MCP transport — POST /slink/<token>/mcp

Streamable-HTTP MCP endpoint scoped to the token's group. Three tools: send_message, steer, get_round. External agents register the URL in their MCP config and talk to the group like any other tool host.

Where it's going

Today, spawning a group is a CLI operation; slink only handles message submission against an already-running group. spec specs/4/N-platform-api.md proposes a unified platform API — POST /v1/groups on gated — that composes with slink so a single round-trip can create a group and return its slink_url + mcp_url ready to use. The slink protocol itself doesn't change; it just gets a sibling group-CRUD surface at the same WEB_HOST.
