arizuko

Multi-tenant Claude agent router. Self-hosted. Channel adapters speak HTTP to gated; per-group Claude Code agents run in Docker containers with an MCP socket bridged in. Go, SQLite (WAL), Docker.

what you get

  • Topics that fork. A conversation can branch into a side-topic; the child inherits its parent’s Claude session via plain cp -r on .claude/. Every new inbound topic also auto-forks from the group’s main topic, so the agent walks in with prior context, not a blank slate. Agents fork themselves with the fork_topic MCP tool. See concepts/topics.
  • Sibling folders share context. A folder’s observe-window messages are visible to its siblings by default; set open=0 on a group to seal it off. Lets corp/eng/sre see what corp/eng/launch-q3 is saying without explicit cross-posts. See concepts/scopes.
  • Slack AI sidebar. The Agents & AI Apps pane is a first-class surface: setStatus, setTitle, setSuggestedPrompts, and pane-context awareness all flow through MCP tools (pane_set_prompts, pane_set_title). The agent knows which channel and thread the pane is attached to. See concepts/slack-pane.
  • Operator dashboard CRUD. dashd serves /dash/ with a routes editor, group settings (open / closed toggle, observe window), and group create / delete. No more editing .env by hand to rewire a route.
  • Topic-suffix routing. target=corp/eng#alerts pins routed messages to a named topic at route time — one topic per source, no manual sorting.
  • Orthogonal primitives. Folders are paths on disk. Routes match on key=glob pairs. ACL grants scope per principal × action × folder. Channels speak HTTP. Containers are per-group. They meet at messages.db and nowhere else.
  • Compose, don’t patch. Adding a channel adapter is a template/services/<daemon>.toml file with a [[proxyd_route]] block. Adding a tenant is a folder + a route + a grant row. New MCP tools land in reference/mcp.
  • Your server, your disk, your backup. One tar of /srv/data/arizuko_<name>/ is the entire instance: messages.db (WAL), group folders, per-user memory, secrets, agent files. No managed control plane.

show me

# seed the instance directory
arizuko create demo
# /srv/data/arizuko_demo/ now has .env, store/, groups/, ipc/, web/pub/

# pick adapters + tokens
$EDITOR /srv/data/arizuko_demo/.env

# register the first group's inbound JID
arizuko group demo add tg:-123456789 main

# render docker-compose.yml from .env + extras, then bring it up
arizuko run demo
# systemd unit: arizuko_demo (sudo systemctl status arizuko_demo)

# talk to the default group from a browser
arizuko invite demo --slink   # prints a token-bound URL

# or from a script — same endpoint, JSON in / JSON out
curl -X POST https://example.com/slink/<token> \
  -H 'Content-Type: application/json' \
  -d '{"text":"hello"}'

architecture in one glance

adapter (teled/discd/slakd/…) --HTTP--> gated
                                          │
                                          ├─ store     (messages.db, WAL)
                                          ├─ container (docker run per group)
                                          │            └─ MCP over unix socket
                                          └─ chanreg   (adapter health, outbound)

timed   — scheduler, writes messages
onbod   — onboarding, OAuth, gated admission
webd    — web chat channel adapter (slink + SSE)
proxyd  — auth-gated reverse proxy, signs identity headers
vited   — static origin: /pub/* + auth-gated default route
davd    — WebDAV workspace (group files)
dashd   — operator dashboards (read-only)

The contract is HTTP plus one SQLite file. gated owns the schema; other daemons connect read/write but never migrate. Adapters, routing table, auth layer, and sandbox are independent components — the daemon ends in d, swaps are local. Full graph and message flow in ARCHITECTURE.md.
