arizuko

self-hosted multi-tenant Claude agent platform · getting started · concepts

Plug Slack, Discord, Telegram, WhatsApp, Mastodon, email, or a web link into arizuko and get a self-hosted Claude agent platform for many groups and many users in one deployment. Each group has its own files, persona, skills, secrets, and routing; each user keeps their own linked identities, grants, encrypted secrets, and memory inside that group.

why

A shared chat agent stops being simple as soon as more than one team and more than one user depend on it. arizuko keeps the control-plane pieces separate: adapters ingest messages, routing maps JIDs to group folders, grants scope what the agent may do, secrets resolve per folder or per user, and agent egress is allowlisted through crackbox. The parts compose without sharing state or policy logic — build the system you need, swap any layer without rewriting the others.

system shape

  • Per-group folder. Each tenant group is a folder on disk with its own persona, skills, diary, files, secrets, and agent session state. The agent runs with that folder mounted as its workspace.
  • Per-user state within a group. One shared agent acts distinctly for Alice and Bob: each user has their own memory file, per-user secrets resolved at tool-call time by a host-side broker (spec 9/11) so the container never holds them, and grant rules layered over the group default.
  • OAuth account linking. GitHub / Google / Discord / Telegram log-ins all resolve to the same user; per-group grants decide what each linked identity can do.
  • Chat-platform adapters. Telegram, Discord, Slack, WhatsApp, Mastodon, Bluesky, Reddit, email, and the built-in slink web chat enter through independent adapter daemons that register over HTTP. Adding a channel doesn’t change routing, grants, or storage.
  • Public web endpoint per group. A token-bound URL anyone can talk to — chat in a browser, or speak the same protocol from a script. Optional MCP transport on the same path. See slink.
  • WebDAV workspace. Mount any group’s files at /dav/<group>/ from Finder, rclone, or curl — the same files the agent reads. webdav.
  • Operator dashboard. /dash/ for grants, routing rules, scheduled tasks, admission queue, and per-group health.
  • Egress isolation by default. Agents run inside crackbox — per-group allowlists decide which hosts a run may reach. Internal services and arbitrary internet egress stay closed unless you open them.
  • Voice in & out. Inbound audio is transcribed (Whisper); the agent can reply with synthesized speech via send_voice (Kokoro TTS).
  • Typed addresses across platforms. One JID shape covers users, chats, posts, threads — routing rules and grants speak the same language regardless of channel.

architecture

Messages arrive through adapter daemons, cross an HTTP boundary into gated, land in one SQLite (WAL) database, then route by JID to a group folder and its Claude runner. Grants, secrets, files, and egress policy are resolved at spawn from that group and, when present, the calling user. Adapters, routing table, auth layer, and sandbox are independent components, not one fused app — the daemon ends in d, the contract is HTTP + SQLite, swaps are local.

operator path

# create an instance and run it
arizuko create demo
cd /srv/data/arizuko_demo
$EDITOR .env             # pick adapters, set tokens
arizuko run demo         # generates compose, brings the stack up

# talk to the default group from a browser:
#   open the slink URL printed in `arizuko invite demo --slink`

# or from a script — same endpoint, JSON in / JSON out:
curl -X POST https://example.com/slink/<token> \
  -H 'Content-Type: application/json' \
  -d '{"text":"hello"}'

One command seeds the instance directory; another renders compose from .env and starts only the daemons you enabled. The deployment stays single-tenant and self-hosted: your storage, your OAuth apps, your bot tokens, your outbound policy.

data and trust boundary

A deployment is one single-tenant instance under /srv/data/arizuko_<name>/: SQLite/WAL state, group folders, per-user memory, encrypted secrets, agent files. Outbound traffic is the chat platforms you enabled (your bot’s replies) and Anthropic (model prompts). Anything else an agent fetches passes through crackbox’s per-group allowlist. Backups are file-level — one tar of the instance directory. There is no managed control plane outside your box.

products

arizuko ships as primitives; the same daemons assemble into different products depending on how you configure groups, channels, and grants.

  • Slack team agent — one shared bot in your team’s Slack workspace, each member gets their own personal files, web workspace, and grants on top of channel-shared context.
