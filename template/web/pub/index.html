arizuko

Multi-tenant Claude agent router. Self-hosted. Channel adapters speak HTTP to gated; per-group Claude Code agents run in Docker containers with an MCP socket bridged in. Go, SQLite (WAL), Docker.

what you get

  • Orthogonal primitives. Folders are paths on disk. Routes match on key=glob pairs. ACL grants scope per principal × action × folder. Channels speak HTTP. Containers are per-group. None of these know about each other — they meet at messages.db.
  • Compose, don’t patch. Adding a channel adapter is a template/services/<daemon>.toml file with a [[proxyd_route]] block. No edit to proxyd/main.go or compose/compose.go. Adding a tenant is a folder + a route + a grant row.
  • Many groups, one process. One instance hosts an arbitrary tree of groups. solo/inbox and corp/eng/sre/oncall run the same code. Each group owns its files, diary, skills, secrets, container, MCP socket.
  • Many users per group, isolated. Within a group, users get per-user memory files and per-user grants. Per-user secrets resolve through a host-side broker at tool-call time — they never enter the container env.
  • Identity unified across channels. GitHub / Google / Discord / Telegram OAuth all resolve to one auth_users.sub. The same person reaching the bot from Discord and Telegram is one principal, one set of grants.
  • Signed identity headers. proxyd is the sole signer; every backend verifies via auth/middleware.go (RequireSigned strict, StripUnsigned lenient). No handler trusts X-User-Sub without a sig check.
  • Per-group sandbox. Each group runs in its own Docker container with bind-mounted group files and a per-group MCP socket. With CRACKBOX_ADMIN_API set, containers attach to an internal: true network; outbound HTTP/HTTPS goes through crackbox, which enforces a per-folder hostname allowlist.
  • Your server, your disk, your backup. One tar of /srv/data/arizuko_<name>/ is the entire instance: messages.db (WAL), group folders, per-user memory, secrets, agent files. No managed control plane.

show me

# seed the instance directory
arizuko create demo
# /srv/data/arizuko_demo/ now has .env, store/, groups/, ipc/, web/pub/

# pick adapters + tokens
$EDITOR /srv/data/arizuko_demo/.env

# register the first group's inbound JID
arizuko group demo add tg:-123456789 main

# render docker-compose.yml from .env + extras, then bring it up
arizuko run demo
# systemd unit: arizuko_demo (sudo systemctl status arizuko_demo)

# talk to the default group from a browser
arizuko invite demo --slink   # prints a token-bound URL

# or from a script — same endpoint, JSON in / JSON out
curl -X POST https://example.com/slink/<token> \
  -H 'Content-Type: application/json' \
  -d '{"text":"hello"}'

architecture in one glance

adapter (teled/discd/slakd/…) --HTTP--> gated
                                          │
                                          ├─ store     (messages.db, WAL)
                                          ├─ container (docker run per group)
                                          │            └─ MCP over unix socket
                                          └─ chanreg   (adapter health, outbound)

timed   — scheduler, writes messages
onbod   — onboarding, OAuth, gated admission
webd    — web chat channel adapter (slink + SSE)
proxyd  — auth-gated reverse proxy, signs identity headers
vited   — static origin: /pub/* + auth-gated default route
davd    — WebDAV workspace (group files)
dashd   — operator dashboards (read-only)

The contract is HTTP plus one SQLite file. gated owns the schema; other daemons connect read/write but never migrate. Adapters, routing table, auth layer, and sandbox are independent components — the daemon ends in d, swaps are local. Full graph and message flow in ARCHITECTURE.md.
