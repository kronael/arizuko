arizuko

a multitenant Claude agent router · getting started · concepts

Plug a chat platform into arizuko — Telegram, Discord, WhatsApp, Mastodon, email, or just a shareable web link — and get a per-group Claude agent with its own files, memory, skills, and scoped permissions. Many groups, many channels, one deployment.

why

Dropping an LLM into a chat is easy. Making it useful for a real team or community isn’t. You want per-user memory, per-group ACLs, scheduled tasks, multi-platform reach, secret management, file workspaces, voice, and you don’t want every agent run with raw network access. arizuko bundles those pieces behind one daemon set and one config file.

what you get

  • Per-group folder. Each group has its own persona, skills, diary, memory, and secrets under one directory. The agent boots with that folder mounted as its workspace.
  • Chat-platform adapters. Telegram, Discord, WhatsApp, Mastodon, Bluesky, Reddit, email, and the built-in slink web chat. Adapters are independent daemons; add one without touching the rest.
  • Public web endpoint per group. A token-bound URL anyone can talk to — chat in a browser, or speak the same protocol from a script. Optional MCP transport on the same path. See slink.
  • Operator dashboard. /dash/ for grants, routing rules, scheduled tasks, admission queue, and per-group health.
  • WebDAV workspace. Mount any group’s files at /dav/<group>/ from Finder, rclone, or curl. webdav.
  • Egress isolation. Agents run inside crackbox — a per-container proxy that only forwards to allowlisted hosts. A prompt-injected agent can’t reach your internal services or the open internet by accident.
  • Voice in & out. Inbound audio is transcribed (Whisper); the agent can reply with synthesized speech via send_voice (Kokoro TTS).
  • Typed addresses across platforms. One JID shape covers users, chats, posts, threads — routing rules and grants speak the same language regardless of channel.
  • OAuth account linking. A signed-in user reaches their groups across providers; per-group grants decide what each identity can do.

show me

# create an instance and run it
arizuko create demo
cd /srv/data/arizuko_demo
$EDITOR .env             # pick adapters, set tokens
arizuko run demo         # generates compose, brings the stack up

# talk to the default group from a browser:
#   open the slink URL printed in `arizuko invite demo --slink`

# or from a script — same endpoint, JSON in / JSON out:
curl -X POST https://example.com/slink/<token> \
  -H 'Content-Type: application/json' \
  -d '{"text":"hello"}'

One create seeds the data dir and a default group. run generates the compose file from .env and brings up only the daemons you asked for. Add adapters by flipping env vars; the rest of the stack doesn’t care.
