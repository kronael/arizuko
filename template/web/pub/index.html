arizuko

Self-hosted orchestration platform for Claude agents. Each team or function is an agent — a folder with its own persona, memory, skills, and channel connections. Agents coordinate, delegate, and commission each other through gated, which brokers every tool call and enforces what each one can reach. No managed control plane, no per-seat pricing. One tar backs up the instance.

why not a shared assistant?

  • Wrong replies land in the thread — one assistant reads sales, support, and ops history from the same memory.
  • Old context changes new work — unrelated threads stay attached unless somebody clears them.
  • Slack becomes the control plane — the real system lives in email, webhooks, and folders, but the assistant only sees one chat tab.
  • Failures are hard to inspect — there’s no file to diff, no route to inspect, and no clean rollback.

arizuko gives each team its own agent, memory, channels, and policy on your own host. Config is files. Rollback is git revert.

how it works

A folder is an agent: a directory with a PERSONA.md, a skills/ set, memory files, and an ACL row. Folders compose into a hierarchy — corp/sales, corp/eng/sre, corp/support — and each one runs in its own Docker container when a message arrives. The container connects back to gated via MCP over a unix socket; every tool call (send a message, read memory, call an API) goes through that socket. The host decides what the agent can reach. All state lives in one SQLite WAL database; the container is stateless. One tar of the data directory is the entire instance.

agents

  • Each folder is an agent. Put teams like corp/sales in their own path with PERSONA.md, skills/, and memory beside them.
  • Identity lives in files. Personality comes from PERSONA.md, tools come from skills/, and memory accumulates across weeks.
  • Topics fork cleanly. When a thread splits, the new topic copies the parent’s session (cp -r .claude/) and both paths run in parallel with full context.
  • Agents can create agents. A root agent can commission child groups on first contact, assign routes, and schedule tasks — using the same MCP tools a human operator would invoke.
  • Knowledge lands where the agent works. Mount the workspace via WebDAV and drop docs, notes, or files. When an agent researches something mid-session, it writes findings to its own facts/ — the next session, it already knows.

channels and coordination

  • Messages arrive through 11 adapters. Telegram, Discord, WhatsApp, IMAP/SMTP, Mastodon, Bluesky, Slack AI sidebar, Reddit, Twitter/X, LinkedIn — each adapter owns its platform connection and delivers events webhooks can’t (threads, reactions, voice messages).
  • Public surfaces are plain URLs. Issue a route token and any folder gets a /chat/<token>/ page (SSE, works in a browser) and a /hook/<token> endpoint (POST JSON, fires the agent).
  • Agents coordinate directly. delegate_group, broadcast, and observe_group are MCP tools — cross-folder work goes through the same controlled socket as everything else.
  • Webhooks become messages; schedules become work. POST alerts, tickets, or CRM events to any folder webhook. Cron tasks per folder run daily summaries, weekly reports, or background jobs with the folder’s full memory.
  • Agents act, not just answer. A curation agent reads Slack intake, decides what matters, and writes a daily brief. A support agent remembers every interaction and escalates without being asked.

operate

  • MCP runs over a gated unix socket. Every agent connects to gated via a unix socket bridged into the container. Tool calls use scoped grants; each folder gets only the access you allow.
  • The operator view lives at /dash/. Routes editor, task scheduler, group settings, secrets, and create / delete — no editing .env by hand to rewire a route.
  • Egress is sandboxed with crackbox. crackbox wraps each container with a per-folder hostname allowlist, HTTPS via CONNECT, and DNS filtering. See crackbox.

start small, grow into it

A minimal deploy is gated and one channel adapter. Add components as you need them — each is a standalone binary with its own TOML block. There’s no managed control plane; every piece is replaceable.

One tar of /srv/data/arizuko_<name>/ backs up the entire instance: messages.db (WAL), group folders, per-user memory, secrets, agent files.

show me

# seed the instance directory
arizuko create demo
# /srv/data/arizuko_demo/ now has .env, store/, groups/, ipc/, web/pub/

# pick adapters + tokens
$EDITOR /srv/data/arizuko_demo/.env

# register the first group's inbound JID
arizuko group demo add tg:-123456789 main

# render docker-compose.yml from .env + extras, then bring it up
arizuko run demo
# systemd unit: arizuko_demo (sudo systemctl status arizuko_demo)

# talk to the default group from a browser
arizuko token issue demo chat   # prints a /chat/<token>/ URL

# or from a script — same endpoint, JSON in / JSON out
curl -X POST https://example.com/chat/<token>/ \
  -H 'Content-Type: application/json' \
  -d '{"text":"hello"}'
