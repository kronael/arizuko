arizuko

Self-hosted platform for running Claude agents at scale. Deploy one agent per team, project, or product — each isolated, persistent, and connected to wherever it needs to be: chat, email, web, webhooks, or APIs. No managed control plane, no per-seat pricing. Runs on a $5 VPS; backs up with one tar; scales to 10,000 teams on the same binary.

why not Claude.ai or the Assistants API?

  • No memory that persists across conversations or users.
  • No per-team isolation — everyone shares one context.
  • No custom channels: you get a chat window, not Slack, email, or webhooks.
  • No self-hosting — your prompts, context, and files go through their servers.

arizuko runs on your infrastructure. The agents accumulate context over weeks, fork topics, publish to the web, and coordinate across teams. That’s the feature, not the complexity.

what you get

  • One agent per team, not one agent for everything. Sales, SRE, and recruiting each get their own container with their own persistent context window. Mixing unrelated concerns in one context produces noise, blended answers, and hallucinated cross-references. Each folder is a clean context boundary that accumulates only the conversations relevant to it.
  • One agent per team, task, department, or product. A support bot, a content pipeline, a PM board, a Slack team agent — each is a folder. Pre-seeded recipes at products/ land one in five minutes.
  • Topics that fork. A side-topic copies its parent’s Claude session via plain cp -r on .claude/, so the agent walks in with context. See concepts/topics.
  • Sibling folders share context. Observed messages in a folder are visible to its siblings unless open=0. See concepts/scopes.
  • Slack AI sidebar. The agent stages suggested prompts and pane titles via MCP and knows which channel the pane is attached to. See concepts/slack-pane.
  • Email in, email out. emaid ingests DMARC-verified mail as a normal channel; unverified senders route to a verb=untrusted quarantine instead of reaching the agent. SMTP send works the same direction.
  • Operator dashboard CRUD. dashd serves /dash/ with a routes editor, group settings, and group create / delete. No editing .env by hand to rewire a route.
  • Everything meets at messages.db. Folders are folders. Routes match on key=glob. ACL is one row: (principal, action, scope, params). Channels speak HTTP. Containers are per-group. No hidden state elsewhere.

start small, grow into it

A minimal deploy is gated and one channel adapter. Add components as you need them — each is a standalone binary with its own TOML block. Nothing coordinates through a managed control plane; every piece is replaceable.

SetupWhat runsBest for
Minimal gated + one adapter First deployment, one team
Standard + dashd + timed Ongoing ops, scheduled tasks
Multi-team + onbod + crackbox Isolated teams, egress control
Product recipe Pre-seeded config slack-team, reality — 5 minutes

One tar of /srv/data/arizuko_<name>/ is the entire instance: messages.db (WAL), group folders, per-user memory, secrets, agent files.

show me

# seed the instance directory
arizuko create demo
# /srv/data/arizuko_demo/ now has .env, store/, groups/, ipc/, web/pub/

# pick adapters + tokens
$EDITOR /srv/data/arizuko_demo/.env

# register the first group's inbound JID
arizuko group demo add tg:-123456789 main

# render docker-compose.yml from .env + extras, then bring it up
arizuko run demo
# systemd unit: arizuko_demo (sudo systemctl status arizuko_demo)

# talk to the default group from a browser
arizuko token issue demo chat   # prints a /chat/<token>/ URL

# or from a script — same endpoint, JSON in / JSON out
curl -X POST https://example.com/chat/<token>/ \
  -H 'Content-Type: application/json' \
  -d '{"text":"hello"}'
