grants

arizuko › reference › grants

What an agent can do at runtime is composed at spawn from three SQLite tables and one folder path. The agent never asks; the rules are pre-baked into its MCP tool manifest. Authorization is the daemon's problem, not the model's.

This page is the syntax + semantics reference. See GRANTS.md for the maintainer-facing walkthrough and worked example. The reference daemon is the in-container MCP server (ipc/ipc.go); the rule engine is grants/grants.go; cross-folder containment lives in auth/policy.go.

1. Tier model

Tier is derived from folder depth, not stored. From auth/identity.go:16:

Tier = min(strings.Count(folder, "/"), 3)
World = first path segment
tierfolder shape (slash count)examplerole
00 slashesroot, maininstance root — unrestricted, CLI-only surface
11 slashatlas/support"world" group — basic send + per-platform actions for every platform routed under this folder's world + management tools + read-write share mount
22 slashesatlas/support/oncalldepth-2 group — basic send + per-platform actions narrowed to routes under this folder's subtree + read-only share mount
33 or more slashesatlas/support/oncall/launch-q3deep group — reply-only (reply, send_file, like, edit)

Why "min(depth, 3)": deeper than three is still tier 3. The default tool surface narrows with depth and stops narrowing at the deepest band.

"World" = first path segment from auth.WorldOf. WorldOf("atlas/support") = "atlas". Independent of tier; used by tier-1 platformRules to pull every JID routed under the world.

2. The four layers

From GRANTS.md § "The four layers":

  • groups table — the folder hierarchy. Source of truth for what exists. store/migrations/0020-groups-refactor.sql:22.
  • user_groups table — (user_sub, folder_glob, granted_at) rows. The folder column is a glob. A row (github:alice, atlas/**) lets alice act on atlas and any descendant. No matching row ⇒ no access. Schema: 0013-user-groups.sql + 0026-user-groups-granted-at.sql.
  • routes table — what JIDs route into which folder. Their presence is what populates platform rules.
  • grant_rules table — per-folder rule overlay (JSON string[]). Appended to tier defaults at spawn. 0006-grant-rules.sql.

3. Rule grammar

Grant rules are strings parsed by grants.ParseRule:

rule    := ["!"] action ["(" params ")"]
params  := param ("," param)*
param   := ["!"] name ["=" glob]
  • ! prefix on the action = deny rule. Last match wins (grants.CheckAction).
  • action is the MCP tool name. Action names match by a glob in which * does not cross a non-word-char boundary (notWordChar). * alone matches every action.
  • param=glob — param's value must match. The glob's * stops at , or ) (isValueDelim).
  • !param — deny when this param is present at all (regardless of value).
  • Bare param (no =) — allow when present, no value constraint.
  • Malformed input (e.g. unterminated paren) parses to Action == "", which matches nothing — a typo cannot silently widen access (grants.go:25-44).

Examples

*                                       # tier-0 instance root
send                                    # any send call
send(jid=telegram:group/*)              # send only to telegram groups
!post                                   # forbid posting
send_file(jid=*)                        # file send to any allowed target
share_mount(readonly=false)             # writable workspace mount
share_mount(readonly=true)              # read-only workspace mount

6. user_groups: who can act on a folder

Glob ACL keyed by user_sub. Match logic in auth.MatchGroups:

  • ** in a row matches any folder (operator).
  • * matches one segment; does not cross /.
  • ** mid-pattern matches zero or more segments (e.g. atlas/** matches atlas AND any descendant).
  • No matching row ⇒ no access. Operator is implicit: a user with a ** row simply matches every folder.
**                  # operator (any folder)
*                   # any single top-level folder (one segment)
pub/*               # any direct child of pub
atlas/**            # atlas and any descendant
atlas/support       # exact-match grant

Tier ≠ ACL. user_groups answers "may sub touch folder F?" Tier answers "what default tools does folder F's agent see?" A user with a ** grant on a tier-3 folder still gets only the deep-group rules; a tier-0 root with no human user_groups rows is still operator-controlled via CLI.

7. Spawn-time composition

From GRANTS.md § "The composition" (canonical site: Gateway.runAgentWithOpts in gateway/gateway.go):

1. Identity  = auth.Resolve(F)
               → {Folder: F, Tier: min(depth, 3), World: firstSeg}

2. Rules     = grants.DeriveRules(store, F, id.Tier, id.World)
               ++ store.GetGrants(F)
               → tier-default list, extended by routed-platform rules
                  and by the per-folder grant_rules row

3. Env       = container.resolveSpawnEnv(store, base, F, J)
               → base ∪ FolderSecretsResolved(F)
                  ∪ UserSecrets(UserSubByJID(J))   (single-user chats only)

The rule list is injected into start.json for the agent and consumed by the in-container MCP server in two places (ipc/ipc.go):

  • grants.MatchingRules(rules, action) at MCP registration — tools with no matching rule are dropped from tools/list entirely. The agent literally cannot see them.
  • grants.CheckAction(rules, action, params) at MCP invocation — a synthesized call still gets refused at the boundary.

8. Worked example

From GRANTS.md § "Worked example". User u:github-1234 sends a Telegram message to chat -123, routed to folder atlas/support.

  1. Inbound admission — proxyd verifies the user's session and stamps X-User-Sub/X-User-Sig. Backend mounts using auth.RequireSigned reject mismatched sigs. For folder access (e.g. dav), auth.MatchGroups(store.UserGroups(sub), folder) must allow atlas/support — **, atlas/**, or atlas/support. No matching row → onbod's admission queue gates the user when ONBOARDING_ENABLED.
  2. Routing — routes resolves a row with match='platform=telegram room=-123', target atlas/support.
  3. Agent spawn — gateway starts a container for folder atlas/support.
  4. Tier — auth.Resolve("atlas/support") → tier 1.
  5. Routed JIDs in world — RouteSourceJIDsInWorld("atlas") collects every <platform>:<room> whose target sits under atlas/.
  6. Rules — grants.DeriveRules(store, "atlas/support", 1, "atlas") yields:
    • send, reply, send_file
    • per-platform: send(jid=telegram:*), post(jid=telegram:*), …
    • schedule_task, register_group, escalate_group, delegate_group, get_routes, set_routes, add_route, delete_route, list_tasks, pause_task, resume_task, cancel_task
    • share_mount(readonly=false)
    • …followed by anything in grant_rules for atlas/support.
  7. Env — resolveSpawnEnv merges FolderSecretsResolved("atlas/support") over the base env (root < atlas < atlas/support, deepest wins). Telegram -123 is a group chat (chats.is_group=1), so no user-secret overlay is added.
  8. MCP gate — IPC server filters tool registration via grants.MatchingRules AND gates each call via grants.CheckAction. Outbound verbs additionally pass through auth.Authorize, which enforces subtree containment.

9. Key invariants

  • Tier ≠ ACL. user_groups (who) is orthogonal to tier (what). A user with a ** grant sees only the tools their folder's tier allows.
  • No silent inheritance of routes. Each folder derives its own rules. Tier-1 worlds see all platforms routed anywhere under themselves; tier-2+ groups see only platforms routed at their own subtree.
  • Route presence gates platform access. No route for bluesky:* to anything under folder F's scope ⇒ no send(jid=bluesky:*) rule ⇒ agent can't post to Bluesky from F even if an adapter is running.
  • Outbound subtree containment is independent of grants. Even a tier-0 root cannot direct-send cross-world via outbound verbs; cross-world traffic goes through escalate_group / delegate_group, each with its own auth.Authorize rule.
  • Secrets ride the same hierarchy as grants but resolve independently. Folder ancestors walk root→F deepest-wins; user overlay applies only when chats.is_group=0. v1 stores plaintext per spec 9/11.
