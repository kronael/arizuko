crackbox

a sandbox proxy for untrusted code · ← hub

what it is

A forward HTTP/HTTPS proxy with a per-source-IP allowlist registry. One binary, one image, one daemon. It takes opaque ids and host lists; it doesn’t know who you are or what you’re running.

You point a workload’s HTTPS_PROXY at it. Egress to listed hosts goes through. Everything else returns 403.

why it exists

Default-deny network is hard to retrofit. You want curl https://github.com from a CI job or AI agent to work, and curl https://evil.com to fail, with the policy decided at spawn time and not embedded in the workload.

iptables rules per process are awkward. Egress firewalls outside the host don’t see who is calling. crackbox sits in between: a tiny proxy that the workload talks to and that knows who the workload is by source IP.

the model

One daemon. One registry: source-IP → (id, allowlist). One matchHost function. That’s the whole mechanism.

  • crackbox proxy serve runs the daemon. Long-lived. Listens on :3128 for forward HTTP and CONNECT-tunneled HTTPS, on :3129 for the admin API.
  • crackbox run --allow X -- <cmd> is convenience sugar: it creates a Docker network, starts the daemon on it, registers one entry for the about-to-run container, then runs your command with HTTPS_PROXY set. On exit, everything is torn down.

Same code path either way. The proxy can’t tell whether the registry holds one entry because a CLI just registered it, or one entry because some long-running consumer happens to have one workload spawned. There is no “single-shot mode” branch anywhere inside the daemon.

try it

docker pull crackbox:latest

# allowed host returns 200
crackbox run --allow github.com -- curl -s -o /dev/null -w '%{http_code}\n' https://github.com

# unlisted host returns 403
crackbox run --allow github.com -- curl -s -o /dev/null -w '%{http_code}\n' https://example.com

# python image, install one package, nothing else
crackbox run --image python:3 --allow pypi.org,files.pythonhosted.org -- pip install requests

Multiple hosts: comma-separated. Subdomains match the parent (--allow github.com covers api.github.com). Empty list (--allow "") blocks everything.

service mode

When you want a long-running proxy that many short-lived workloads register against, run the daemon directly:

crackbox proxy serve --listen :3128 --admin :3129

Then drive the registry over the admin API:

  • POST /v1/register — body {"ip": "...", "id": "...", "allowlist": ["host1", "host2"]}
  • POST /v1/unregister — body {"ip": "..."}
  • GET /v1/state — current registry snapshot
  • GET /health — liveness; performs a TCP self-test against the proxy listener and returns 503 {"status":"proxy_down"} if the proxy port is dead

Optional bearer-token auth on the mutating endpoints via CRACKBOX_ADMIN_SECRET. Read-only /v1/state and /health stay open. Empty secret leaves mutations unauthenticated and logs a warning at startup.

Optional disk persistence via CRACKBOX_STATE_PATH. Each register / unregister rewrites the file atomically (tmp + rename). Startup reloads it. Corrupt or missing file resets to empty with a warning — a stale snapshot can never block startup. Empty path keeps the registry RAM-only.

transparent mode

Same daemon, second way to receive traffic. Forward mode wants the client to set HTTPS_PROXY; transparent mode wants the client side to iptables REDIRECT port-80/443 traffic to crackbox. The daemon reads the pre-NAT destination via SO_ORIGINAL_DST, peeks SNI (443) or Host: (80), runs the same per-source-IP allowlist, splices on success.

The transparent listener is bound by default on :3127. Idle when nothing redirects to it — binding costs nothing. Disable with --transparent "" or set transparent_listen = "" in the config file. Linux only; v1 handles ports 80 and 443.

sudo iptables -t nat -A OUTPUT -p tcp --dport 443 \
  -m owner --uid-owner tester \
  -j REDIRECT --to-ports 3127

Same registry, same matchHost, same Allow(). Two listeners is just two ways into one daemon.

configuration file

Optional TOML, looked up in order: --config path, $XDG_CONFIG_HOME/crackbox/crackbox.toml, ~/.crackboxrc, /etc/crackbox.toml. Missing file is fine — defaults apply. Precedence: flags > env > config file > defaults.

[proxy]
listen = ":3128"
admin_listen = ":3129"
transparent_listen = ":3127"   # set to "" to disable

[admin]
secret = ""                    # bearer token; empty disables auth

[state]
path = ""                      # registry persistence; empty = RAM-only

what gets matched

Host name only. The matcher is exact, subdomain-aware, case-insensitive, and strips a trailing dot. Nothing else.

  • no IP-level rules
  • no path-level rules
  • no per-method rules
  • ports 80 and 443

One pure function, three lines. That’s the entire policy surface.

what it doesn’t do

  • no TLS interception or MITM
  • no secret injection into outbound requests
  • no response scanning or content filtering
  • no traffic auditing or per-request logs
  • no runtime allowlist mutation (crackbox allow / deny)
  • no supervision, idle-shutdown, or auto-restart — lifecycle belongs to Docker compose or systemd

The single mechanism is small on purpose. Everything above is a separate concern that you compose with a separate tool.

threat model

Non-cooperating clients fail closed. crackbox run puts the workload on a Docker network with no default route; clients that ignore HTTPS_PROXY simply can’t reach anything off-network. A workload can’t escape its allowlist by setting HTTPS_PROXY="" — there’s no other path out.

Most tooling honors proxy environment variables natively: curl, wget, pip, npm, git, go, apt. Node’s fetch needs a one-line shim to set the global agent. That’s the only common gotcha.

Trust assumption: source IP within the daemon’s network is honest. The network topology has to be such that only your workload can reach the proxy on its registered IP. crackbox run arranges this for you with a private Docker network. In service mode you arrange it yourself.

install

docker pull crackbox:latest

One image, ~15 MB. Daemon RAM stays around 15–20 MB regardless of registry size. crackbox run adds one user container plus a Docker network, around 10 MB extra and roughly half a second of spawn latency.

uninstall

docker rmi crackbox:latest
docker network ls | grep crackbox- | awk '{print $1}' | xargs -n1 docker network rm
