wire webhooks into arizuko

GitHub, Linear, Sentry, Stripe, Grafana — every SaaS with a "send a POST when X happens" knob can hand events to an arizuko agent today. No new daemon: webd's /slink/<token> endpoint already accepts POSTs and routes them through gated like any other inbound message.

how it works

  1. Every group has a slink_token in its row (auto-generated at group creation by onbod/main.go:498).
  2. POST {"content": "...", "topic": "..."} to https://<WEB_HOST>/slink/<token>.
  3. webd looks up the token, stores the message with chat_jid = web:<folder> and sender = anon:<sha-of-ip>, then forwards to gated — same path a slink chat user takes.
  4. Gated runs the routing table against web:<folder>; the agent for that folder spawns and processes the turn.

JID shape today

Every webhook POST through one token lands with the same chat_jid — the group's folder: 

chat_jid:  web:<folder>
sender:    anon:<sha-of-ip>
source:    web

The token itself is not in the JID. There is one token per group folder (store/groups.go:193: GroupBySlinkToken is a unique lookup). To distinguish "this came from GitHub" from "this came from Stripe", scope each source to its own group folder and give each its own token.

topic as a within-source label

Inside one folder, set topic in the POST body to fan events into threads: github/issues, github/prs, stripe/charges. The gateway treats topic as the conversation key, so the agent sees one thread per event-class without polluting the chat_jid.

get a token

Create a folder for the source (if you don't already have one), then read its token from the DB: 

# create the folder (the JID is the route key gated will match on)
arizuko group acme add 'web:github' github

# read its slink_token
sqlite3 /srv/data/arizuko_acme/store/messages.db \
  "SELECT slink_token FROM groups WHERE folder='github'"

Treat the token like a password — whoever holds it can POST messages into that folder.

example: GitHub → arizuko

  1. In the repo settings → Webhooks → Add webhook.
  2. Payload URL: https://<WEB_HOST>/slink/<token>
  3. Content type: application/json
  4. SSL verification: enabled.
  5. Events: pick what you want the agent to hear about (issues, pull requests, releases, …).

GitHub's payload field names don't match what webd expects. webd reads content and topic; GitHub sends action, issue, pull_request, and so on. Two ways to bridge:

  • Adapter script in front of the webhook. A tiny script (Cloudflare Worker, Lambda, or a Python one-liner on the same host) accepts GitHub's payload, extracts a one-line summary, POSTs {content,topic} to slink. Honest path.
  • Skip the bridge for sources that send plain text. Grafana alerts, Healthchecks.io, custom curl-from-cron all let you write the JSON body directly.

example: Grafana alert → arizuko

Grafana's Webhook contact point sends a JSON body you control. Set the template to: 

{
  "content": "[{{ .Status }}] {{ .CommonLabels.alertname }} — {{ .CommonAnnotations.summary }}",
  "topic": "grafana/{{ .CommonLabels.alertname }}"
}

Each distinct alert keeps its own thread; firing/resolved transitions land in the same one.

example: Stripe → arizuko

Same shape as GitHub — Stripe's payload is JSON but not in webd's {content, topic} shape. Use a thin adapter to project the event to a one-liner like "charge.succeeded $42.00 cus_ABC" and POST it.

Stripe also sends a Stripe-Signature HMAC header. The adapter is where you verify it — arizuko does not (see security below).

security: what arizuko does and doesn't do

No HMAC verification on inbound slink POSTs. The slink token is the credential. If the token leaks, anyone who finds it can POST messages into that folder until you rotate. webd accepts JSON (webd/slink.go:194) or form-encoded bodies and never checks a signature header.

The honest mitigations:

  • One token per source. A leaked GitHub-source token can post into the github folder. It cannot post into stripe or any other folder.
  • Scope the folder narrowly. Grants on the target folder still apply to the agent's tool use. A webhook-only folder needs no human grants, so an attacker posting in cannot use the agent to reach into other folders — the agent's delegate_group and escalate_group targets are constrained by the folder's grants (grants).
  • Rotate. Replace the token in the DB and update the webhook URL at the source. There is no revoke CLI today — UPDATE the row.
  • Verify upstream signatures in an adapter. If the source ships an HMAC header (Stripe, GitHub, Slack-style), check it in the adapter script before POSTing to slink. arizuko sees only verified events.

rate-limiting

Two backstops live in webd. POST body is capped at 64 KiB (maxFormBody, webd/server.go:25). The topic field is capped at 128 chars (maxTopicLen, same file) to keep the SSE hub's map size bounded. Beyond that, the platform does not rate limit a hot webhook source. A noisy upstream becomes a noisy agent turn queue. Use folder-scoped routes with #observe mode (routing) to land events without firing the agent on every one.

how this looks to the agent

The agent sees a normal message in its folder, sender anon:<hex>. If you set topic, the agent sees that as the thread. To respond, the agent calls reply with the same chat_jid — the reply lands on webd's SSE stream (webd), reachable by anyone subscribed to that folder via the chat widget or an SDK call. The agent's reply does not go back to the upstream webhook source — webhooks are one-way ingest.

checking it works

# tail logs as you fire a test event
sudo journalctl -u arizuko_acme --since "30s ago" -f | grep -iE 'slink|web:'

Look for slink inbound folder=github when the POST lands, then spawn folder=github as the agent starts a turn. A 404 means the token is unknown (typo or rotated). A 400 means the body is missing content.

future enhancement (not shipped)

Operator-tunable JID prefix per token. Today every webhook through a folder's token lands as web:<folder>. A small change to webd/slink.go:372 (injectSlink) plus a new jid_prefix column on the groups table would let you label a token "github" and have inbound post as webhook:github/<folder>. Useful for route rules that fan one folder across many sources. Not shipped — for now, one folder per source covers the same intent. Open an issue if you need the JID-level split.
