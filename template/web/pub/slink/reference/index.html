token design

  • 32 random bytes, base64url-encoded (~43 chars, 256 bits).
  • Public by design — freely shared in page source, bookmarks, scripts.
  • Generated once at group registration, never rotated.
  • Lives in groups.slink_token; proxyd resolves it on every /slink/* request.
  • Security comes from rate limiting, not secrecy. Treat a leaked token like a leaked rate-limit bucket, not like a leaked password.

Tunable via SLINK_ANON_RPM / SLINK_AUTH_RPM in the instance .env.

sender identity

  • Valid JWT → sender = jwt.sub, sender_name = jwt.name.
  • No Authorization header → sender = anon:<ip-hash>, no name.
  • Malformed JWT → 401. Omitting the header is fine; sending a broken one is not.

round-handle protocol

A round is one container spawn: exactly one inbound batch in, exactly one submit_turn out. The protocol exposes the round as a first-class object with a stable handle so a caller can wait on it, page through frames, or stream them over SSE.

turn_id = the originating inbound messages.id. That id is the round handle — the caller doesn’t mint it; the POST response hands it back.

POST

Body: content=<text>&topic=<optional> (form) or {"content":"...","topic":"..."} (JSON).

Default response (JSON):

{
  "user": { "id": "msg_abc", "content": "...", "created_at": "..." },
  "turn_id": "msg_abc",
  "status": "pending"
}

turn_id == user.id for the first message of a round.

Content negotiation:

  • Accept: text/html — legacy HTMX fragment used by the browser chat widget.
  • Accept: text/event-stream — legacy SSE upgrade keyed on (folder, topic); preserved for the chat widget. New clients should POST normally and then open /turn/<id>/sse on the round handle.

GET turn snapshot / cursor

GET /slink/<token>/turn/<id>
GET /slink/<token>/turn/<id>?after=<msg_id>

Returns:

{
  "turn_id": "msg_abc",
  "status": "pending|done|failed",
  "frames": [
    {"id":"out_001","content":"...","created_at":"...","kind":"message"},
    {"id":"out_002","content":"...","created_at":"...","kind":"status"}
  ],
  "last_frame_id": "out_002"
}

frames[] is ordered by created_at ascending; each id is a stable messages.id. Use it as ?after=<id> on the next call to fetch only new frames.

kind:

  • message — the agent’s reply.
  • status — an interim <status> block from the agent’s status protocol (rendered as the ⏳ prefix on the platform side).

Stop polling once status == "done". "failed" carries an error field with a short message.

GET status (cheap)

GET /slink/<token>/turn/<id>/status
{
  "turn_id": "msg_abc",
  "status": "pending",
  "frames_count": 2,
  "last_frame_id": "out_002"
}

Use this to decide between “poll once and stop” vs “open SSE” without paying for the frame payload.

GET SSE stream

GET /slink/<token>/turn/<id>/sse

Events:

  • event: message / event: status — same payload shape as a snapshot frame, one per <status> block or final reply.
  • event: round_done — fires exactly once after submit_turn. Payload {turn_id, status, error?}. The server then closes the stream.

Last-Event-Id reconnect is supported. On reconnect the server replays frames whose id is greater than the last seen id (same query as ?after=<id>) before resuming live.

steering

POST /slink/<token>?steer=<turn_id>   body: content=...

Inject a follow-up tied to an already-issued round. Two outcomes, both HTTP 200:

  • Round still pending. The Claude Agent SDK does not support mid-turn input injection — the in-flight LLM context is frozen until the current turn produces its final response. So “steering” is semantically “cut to the front of the next-turn queue”: per-folder serialization makes the steered message the immediate next round. Response carries chained_from: "<original_turn_id>".
  • Round already done. The steer becomes a fresh round; chained_from is omitted.

Either way, observe progress through the same handle endpoints on the new turn_id.

browser chat UI (legacy path)

The widget served at GET /slink/<token> uses the legacy POST /slink/<token> (HTML fragment) + GET /slink/stream SSE combo, keyed on (folder, topic) rather than on a turn handle. It continues to work and is unchanged. New consumers should use the round-handle endpoints; the widget will migrate when there’s a UX reason.

hub keying

The webd SSE hub stays keyed by (folder, topic). The round-handle SSE handler subscribes to that key and filters frames whose payload turn_id matches the requested round. The gateway notifies webd via POST /v1/round_done when submit_turn lands; webd publishes a round_done frame on (folder, topic) so any open round-handle SSE subscriber can disconnect cleanly.

schema dependencies

  • messages.turn_id (migration 0038, indexed) — every outbound assistant message produced inside a run carries the inbound message id that triggered it. Paging is exact via foreign key, not a time-window correlation.
  • turn_results(folder, turn_id) (migration 0036) — one row per completed round; consumed by the snapshot and status endpoints.

retention

Rounds are durable until the underlying message is pruned. Today messages and turn_results are kept indefinitely; when a retention policy is added it covers both tables uniformly.

operator CLI

arizuko send <instance> <folder> "<msg>" [--wait | --stream | --steer <turn_id>]
arizuko send <instance> <folder> --stdin            # read body from stdin
  • --wait — poll /turn/<id>?after=<last> until status != "pending"; print frames as they arrive.
  • --stream — same, but over /turn/<id>/sse; closes on round_done.
  • --steer <turn_id> — POST with ?steer=<turn_id>.

Reads slink_token from the instance store and WEB_HOST from the instance .env; posts via slink. No new transport, no new auth.

MCP transport note

POST + SSE on /slink/<token> is structurally the deprecated MCP SSE transport (v2024-11-05). A future variant could expose a standards-compliant MCP Streamable HTTP endpoint for native Claude Desktop connection without changing the round-handle semantics.
