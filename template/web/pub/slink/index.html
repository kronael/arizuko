slink

a public HTTP endpoint for every group · ← hub

Every group has a public URL. POST a message to it and the group’s agent runs — one container spawn, one reply. Watch the round finish by polling, by streaming SSE, or by ignoring it. Same endpoint backs the browser chat widget, curl calls from a cron job, and the arizuko send CLI.

why

An agent is only useful if you can reach it from where the work happens: a web page, a shell script, a CI job, a Lambda. Slink is the lowest common denominator: a token in a URL, a POST with text, an HTTP response carrying a handle for the round that just started.

No SDK to install, no auth dance for the simple case, no separate streaming protocol. Plain HTTP. Public-by-design tokens with rate limiting instead of secrecy.

show me

# post a message, get back a turn_id (the round handle)
curl -X POST https://<host>/slink/<token> \
     -d 'content=summarize today's diary'
# {"user":{"id":"msg_abc",...},"turn_id":"msg_abc","status":"pending"}

# stream the reply over SSE; closes on round_done
curl -N https://<host>/slink/<token>/turn/msg_abc/sse

# or: poll for new frames since the last one you saw
curl https://<host>/slink/<token>/turn/msg_abc?after=out_002

Tokens are 32 random bytes, base64url. They live in the URL on purpose — share the link, paste it into a bookmark, embed it in a script. Rate limits keep anonymous traffic bounded; a JWT raises your bucket.

good for

  • Web chat widgets. The browser UI at /slink/<token> is a slink client. Drop the same token into your own page and you’ve got a chat surface.
  • Cron jobs. One curl, one summary in the group’s diary. arizuko send <inst> <folder> "..." is a 30-second wrapper.
  • CI notifications. POST a build result; the agent decides whether it’s worth waking anyone up.
  • Scripted multi-step flows. POST, watch the round, decide, POST a follow-up with ?steer=<turn_id> to chain it onto the same chat.
  • Sync-on-result calls. --wait or SSE turn the agent into a request/response service for callers that need an answer before they continue.

two shapes

Fire-and-forget. POST and walk away. The response carries the turn_id if you want to look up the round later. This is the cron-job and notification shape.

Wait-for-round. POST, then either poll GET /turn/<id>?after=<last_id> for incremental frames, or open GET /turn/<id>/sse and read until the terminal round_done event closes the stream. Same handle, two transports — pick whichever fits your runtime.

where do I start

Operator path: arizuko send <instance> <folder> "<msg>" [--wait | --stream | --steer <turn_id>]. Trivial wrapper over the protocol — reads the folder’s slink_token from the instance store, posts, optionally streams. Nothing magic.

Integration path: any HTTP client. The reference covers token issuance, the round-handle endpoints, SSE event shapes, steering semantics, the rate-limit tiers, and the schema rows the protocol relies on.
