# Discoverable OpenAPI for every daemon

**Current state (2026-05-13):** 0 daemons implement this. No
`/openapi.json` handler is registered anywhere in the codebase
(`grep -rn openapi --include='*.go'` returns no hits). Phased rollout
lives in [`specs/6/00-finalise-plan.md`](00-finalise-plan.md).

Every arizuko daemon serving HTTP ships a generated OpenAPI 3.1 document
at `GET /openapi.json` and a Swagger UI page at `GET /docs/`. Generated
from Go types, not hand-written, so it cannot drift from the handler. A
top-level aggregator (proxyd-served `/docs/`) renders every daemon's
spec in one Swagger UI.

Sibling to [R-platform-api.md](R-platform-api.md): that spec defines the
`/v1/*` contract; this one says the contract must be discoverable in the
same way HTTP itself is.

## Why

The `/v1/*` shape is the platform's external surface — dashboard, agent
MCP federation, third-party operators, and code generators
([`specs/10/2-printing-press.md:124-136`](../10/2-printing-press.md))
all consume it. Today the surface lives in Go source
([`api/api.go:50-56`](../../api/api.go),
[`webd/server.go:48-78`](../../webd/server.go),
[`dashd/main.go:123-134`](../../dashd/main.go),
[`onbod/main.go:79-89`](../../onbod/main.go),
[`ttsd/main.go:46-49`](../../ttsd/main.go)) plus prose in each
`<daemon>/README.md`. That doesn't scale: handlers churn, READMEs lag,
no consumer can introspect the platform at runtime.

[R-platform-api.md](R-platform-api.md) Phase 2 already calls for
per-daemon `openapi.json`; this spec is the contract for how they're
produced, where they live, and how drift is prevented.
[`specs/10/2-printing-press.md:130-134`](../10/2-printing-press.md)
already assumes `GET /v1/openapi.json` exists — **that path is wrong**
and is reconciled below.

OpenClaw managed-agents ships one hand-written
[`refs/openclaw-managed-agents/openapi/openapi.yaml`](../../refs/openclaw-managed-agents/openapi/openapi.yaml)
(710 lines, 21 paths). That's the shape we want, but hand-written is
the failure mode — they edit it by hand and it drifts. Generation is
the upgrade.

## Mandate

- Every daemon serving HTTP exposes `GET /openapi.json` returning
  valid OpenAPI 3.1 for its own routes.
- Every daemon serving HTTP exposes `GET /docs/` rendering Swagger UI
  against its own `/openapi.json`.
- The document is **generated** from Go types + handler registration.
- Each daemon commits `<daemon>/openapi.json` to the repo for PR diff
  visibility.
- Every channel adapter's `server.go` does register at least
  `GET /files/` + `GET /health` (see
  [`slakd/server.go:40-42`](../../slakd/server.go),
  [`discd/server.go:27`](../../discd/server.go),
  [`mastd/server.go:31`](../../mastd/server.go),
  [`teled/server.go:40`](../../teled/server.go)) so each ships its own
  `/openapi.json`. The `chanlib.RouterClient` _consumer_ contract gets
  one combined `chanlib/openapi.json` documenting the calls every
  adapter makes into `chanreg` ([`api/api.go:50-55`](../../api/api.go)).

Path is `/openapi.json`, not `/v1/openapi.json` — the doc spans all
versions a daemon serves, it's infrastructure not a versioned
resource. Reconciles
[`specs/10/2-printing-press.md:130-134`](../10/2-printing-press.md).

## Tooling: huma

Pick **[huma](https://huma.rocks)**. Handlers are typed Go functions;
the OpenAPI document falls out of registration. No drift between
annotation comments and code, no separate schema file to maintain.

Alternatives considered:

- `swaggo/swag` — annotation comments above handlers; annotations
  drift and the reviewer can't tell at a glance whether they match
  the handler. Rejected.
- `kin-openapi` — schema-first; doesn't generate paths from
  registration, handler-side errors aren't compile-checked. Useful
  as a CI validator library, not as the primary.
- `oapi-codegen` — codegen both directions; forces a particular
  handler signature, heavier than arizuko's small surfaces need.
- Hand-write JSON — what we're avoiding. OpenClaw's 710-line yaml
  is the cautionary tale.

Legacy non-`/v1/*` handlers (slink HTML, dashd HTMX, webd `/send`,
ttsd reverse-proxy) stay un-instrumented; their `/openapi.json`
lists them as opaque routes (path + method + 200, no schema). They
migrate to huma when they graduate to `/v1/*`.

## File layout (per daemon)

```
<daemon>/
  openapi.go          # huma API registration
  openapi.json        # committed; regenerated by `make openapi`
  openapi_test.go     # parse + route-coverage validity
```

Plus one repo-wide `openapi/index.html` for the aggregator Swagger UI
that loads every daemon's spec via the Servers dropdown. proxyd
routes top-level `/docs/` to it.

## Discoverability

- `EXTENDING.md` extension-points table gains one row: a new daemon
  registers via `openapi.go` and exposes `/openapi.json` + `/docs/`.
- Each daemon's `README.md` gains a Docs row pointing at
  `http://<daemon>:<port>/docs/` and `http://<host>/docs/`.

## CI guarantee

- `make openapi` regenerates every committed `<daemon>/openapi.json`.
- CI runs `make openapi && git diff --exit-code` — any drift fails.
- Each `openapi_test.go` validates: (1) spec parses as OpenAPI 3.1
  (kin-openapi validator); (2) every documented route is hit by an
  integration test (grep tests for the path string — coarse, cheap).
- Pre-commit hook calls `make openapi` on touched daemons.

## Phasing

1. **Phase 1 — daemons with stable HTTP surface.** `gated` (largest;
   per [R-platform-api.md §Daemon ownership](R-platform-api.md)),
   `proxyd`, `dashd` (document the API the dashboard _consumes_, not
   its HTML), `webd`, `onbod`, `timed`, `davd`. Order by
   R-platform-api Phase 2-3; each daemon documents itself as it grows
   its `/v1/*`.
2. **Phase 2 — `chanlib` combined doc.** Consumer contract for the
   `chanreg`/router calls every adapter makes
   ([`api/api.go:50-55`](../../api/api.go)).
3. **Phase 3 — `ttsd`.** Single-endpoint proxy
   ([`ttsd/main.go:46-49`](../../ttsd/main.go)). Trivial, last.

whapd is TypeScript ([`whapd/`](../../whapd/)); same convention,
TS tooling — emit `whapd/openapi.json` from handler types, expose
`/openapi.json` + `/docs/`.

## Out of scope

- GraphQL / gRPC — HTTP/JSON only.
- SDK generation — `specs/10/2-printing-press.md` handles that.
- Per-tenant API surface variants — one shape per daemon.
- MCP tool introspection — `tools/list` covers it; MCP is JSON-RPC
  over a unix socket, not HTTP.
- Documenting unauthenticated HTML pages (`/pub/*`, `/slink/{token}`)
  — not API surface; noise without consumer value.

## Acceptance

- `curl http://localhost:<port>/openapi.json` returns valid OpenAPI
  3.1 for every Phase 1 daemon.
- `curl http://localhost:<port>/docs/` renders Swagger UI per daemon.
- `curl http://<host>/docs/` renders the aggregator (every daemon's
  spec via Servers dropdown).
- `make openapi && git diff --exit-code` clean on CI.
- For at least one daemon (target: `timed`, smallest `/v1/*` surface),
  every route in `main.go` appears in `openapi.json` and is hit by an
  integration test.

## Open

- **Aggregator hosting.** proxyd front door, or a dedicated webd
  endpoint? Lean proxyd — already routes every daemon. Decide at
  Phase 1.
- **Auth on `/openapi.json`.** Public, or `Bearer`-gated? Lean
  public — the schema is not a secret; OpenClaw matches
  ([`openapi.yaml` `security: []`](../../refs/openclaw-managed-agents/openapi/openapi.yaml)).
- **Swagger UI vs Redoc.** Lean Swagger UI (try-it button matters
  for operators). Redoc later if a marketing page emerges.
