# specs/5 — platform core: surfaces, identity, routing, tenancy, runtime

The capabilities layer. Cross-cutting platform specs that aren't
tied to a single channel adapter:

- **Surfaces** — MCP, REST, web, voice, WebDAV, openapi discovery
- **Identity** — auth, ACL (unified into one `Authorize`), JID
  format, multi-account, operator
- **Routing** — route table, topics, engagement, mention promotion,
  webhook routes
- **Tenancy** — org-chart self-service, invites, secrets layering,
  user-spawned agents, daemon genericization
- **Runtime** — agent pipeline, middleware, modality envelope,
  output styles, message actions

## Where this leads

Phase 5 capabilities are the foundation for phases 6 + 7:

- [6/](../6/) layers enterprise hardening (encryption, audit, SSO,
  secret broker, MITM egress) on top of phase 5's surfaces +
  identity + tenancy.
- [7/](../7/) closes and minimizes: `7/1` finishes the unification
  begun in `5/5-uniform-mcp-rest.md`; `7/2` sharpens the data
  entities defined here; `7/3` moves the cold tier into git.

The BYOA unlock for phase 7 is **`32-tenant-self-service.md` Phase C**
(folder/user-scope secrets layering) — that work lands here and
unblocks the git-as-truth move.

## Scope boundary

Per-channel adapter behavior (Slack/Telegram/Discord-specific
quirks) lives next to the daemon code in `teled/`, `slakd/`, etc.,
not in spec files. Phase 6 historically picked up channel-flavored
items (Slack agent pane, multi-workspace) — those are exceptions,
not the theme.
