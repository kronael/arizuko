ARIZUKO

microservice gateway — 12 daemons, 7 channel adapters, MCP over unix socket
Multitenant Claude agent gateway. Channel adapters forward messages via HTTP; gateway routes to containerized agents; agents respond via MCP tools.
蟻塔女 — mistress of the ant hill · 30+ packages, 180+ tests, compose-deployed
Architecture: microservices. All daemons are separate binaries coordinating via a shared SQLite DB (WAL mode). Only gated + at least one channel adapter are required. Everything else is optional. Generated via arizuko generate <instance> → docker compose up.
TL;DR
Channel adapters (7 platforms) POST messages to the router HTTP API. Gateway polls SQLite, applies impulse-gate batching, resolves routing rules, and spawns containerized Claude agents via docker. Agents communicate back via MCP over unix socket (tools filtered by grants engine). Output streams back through the adapter's /send endpoint. Shared SQLite WAL, no other external dependencies beyond docker. 30+ packages, 180+ tests, 12 daemons.

Message Flow

// Adapter → HTTP API → store → gateway poll → impulse gate → queue → container → adapter

Channel adapter → POST /v1/messages (api) → store.PutMessage
  → gateway.messageLoop (polls every 2s)
  → impulseGate.accept(jid, msgs) (weight-based batching; social verbs=0, threshold=100, max_hold=5min)
  → checkTrigger (direct mode or @name regex)
  → handleCommand (/new, /ping, /chatid, /stop)
  → prefix dispatch (@name → named group, #topic → topic session)
  → router.ResolveRoutingTarget (delegate to child group if matched)
  → queue.EnqueueMessageCheck → processGroupMessages
    → router.FormatMessages (XML batch, observed JIDs as <observed>)
    → grants.DeriveRules → inject into start.json
    → container.Run (docker run -i --rm)
    → stream output → router.FormatOutbound
    → HTTPChannel.Send → POST /send to channel adapter

Each agent runs in its own docker container. The gateway mounts group folders, IPC dirs, and session state. Agent reads a JSON prompt from stdin, writes JSON to stdout delimited by ---ARIZUKO_OUTPUT_START--- / ---ARIZUKO_OUTPUT_END---. MCP tools available over a unix socket (ipc/<folder>/gated.sock) bridged in via socat.

Impulse Gate

Messages accumulate per JID with a weight score. Agent fires when the score reaches 100 or after 5 minutes (max-hold flush). Social verb events (join/edit/delete) carry weight 0 — they don't trigger agents alone. Per-JID config lives in routes.impulse_config JSON column. Platform-level wildcard: a config keyed discord: applies to all Discord JIDs.

Observed Messages

Non-trigger JIDs routed to the same group appear as <observed> in the agent prompt. Enables patterns like "watch Discord, respond via Telegram." All messages are merged into one <messages> block sorted by timestamp.

IPC: MCP over Unix Socket

The ipc/ package starts one MCP server per group before container spawn, listening on ipc/<folder>/gated.sock (mark3labs/mcp-go library). socat bridges the socket into the container. Agent-runner configures the arizuko MCP server (mcp__arizuko__* tool names) in settings.json. Tools exposed are filtered by the grants engine for the caller's group. Socket is cleaned up after the container exits.

Grants Engine

Rule strings: [!]action[(param=glob,...)]. Evaluation is last-match-wins; no match = deny. NarrowRules for parent→child delegation (child can only narrow, never widen). DeriveRules computes defaults by tier: tier-0 gets *, tier-1 gets platform send actions + management tools, tier-2 gets send only, deeper gets send_reply only. Rules are injected into start.json at container spawn time.

Auth System

JWT + 30-day rotating refresh tokens. OAuth: Google, GitHub (optional org gate), Discord. Telegram Login Widget. Login rate limiting: 5 attempts per 15 min per IP. Secure cookie flag derived from HTTPS URL. Telegram replay protection (5-minute auth_date window). Operator (no group restriction) vs. user (restricted to user_groups table rows) planes.

Web Channel

web:<folder> JIDs. Messages arrive via proxyd (HTTP/SSE), stored in shared SQLite. Gateway discovers active web JIDs via store.ActiveWebJIDs. Each web message carries a topic; processWebTopics runs one agent per topic. Slink tokens for public embeddable widgets, rate-limited at 10 req/min per IP. WebDAV (WEBDAV_ENABLED=true): proxyd exposes /dav/ as an auth-gated reverse proxy to a davd container mounting groups/ read-only.

Mythology

蟻塚女 — mistress of the ant hill. Ants embody organized persistence: each carries its fragment without understanding the whole, and the colony builds anyway. The mistress does not command — she coordinates by structure: routing rules determine which ant receives the work, grants determine what each ant can touch, the circuit breaker determines when to stop and wait. Every session a grain is added. Every diary entry is the colony remembering. Every ant is expendable; the hill is not.

Setup Profiles

Three deployment profiles from smallest to full. All share the same SQLite DB and docker infrastructure.

Profile 1 — Nanoclaw mode
1
gated only — one bot, one agent, one channel
Closest to the original nanoclaw: no compose, no dashboard, no auth gate. Run the gated binary directly with Telegram configured. One group, direct mode, agent responds to all messages.

Required env: TELEGRAM_BOT_TOKEN, CONTAINER_IMAGE=arizuko-agent:latest, AUTH_SECRET.
No WEB_HOST, no OAuth, no WebDAV, no onboarding.
The teled adapter daemon registers with gated on startup; or run gated standalone with the legacy channel path.
Profile 2 — Operator setup
2
gated + teled + proxyd + dashd
Adds the operator dashboard and auth gate. Login via Google or local credentials. /dash/ shows agent status, activity, tasks, memory browser.

Required env: TELEGRAM_BOT_TOKEN, CONTAINER_IMAGE, AUTH_SECRET, GOOGLE_CLIENT_ID, WEB_HOST=https://yourhost.com.
Generate and run: arizuko generate <instance> → docker compose up.
No social adapters, no WebDAV, no onboarding.
Profile 3 — Full stack
3
all services + social adapters + WebDAV + onboarding
Full compose stack. All channel adapters, onboarding state machine, WebDAV file browser, web chat.

Key env additions: ONBOARDING_ENABLED=true, WEBDAV_ENABLED=true, ONBOARDING_PLATFORMS=telegram,whatsapp (filter which platforms trigger onboarding), plus adapter-specific tokens (DISCORD_BOT_TOKEN, MASTODON_ACCESS_TOKEN, etc.).
Compose includes onbod (auto, when ONBOARDING_ENABLED=true) and davd (dufs, when WEBDAV_ENABLED=true). Social adapter service files in template/services/ are drop-in toml includes.

Channel Adapters

Each adapter is a standalone daemon that registers with the router via POST /v1/channels/register. Router pings GET /health every 30s; three failures trigger auto-deregister. All Go adapters share chanlib/ for HTTP client, auth middleware, and inbound message types. Messages carry a verb field (default ""; social platforms set join/edit/delete on non-standard events).

teled — Telegram shipped
Long polling via go-telegram-bot-api. Markdown to HTML conversion.
JID: telegram:<chatID>. @bot mention translation. Media placeholders (photo, video, voice, sticker, document, location, contact). File sending routed by extension. 4096 char limit with HTML retry fallback. Forward/reply metadata. Reply-to threading with chunk chaining.
discd — Discord shipped
WebSocket gateway via discordgo. Intent-based event handling.
JID: discord:<channelID>. Bot mention <@id> to @name translation. Attachment type detection. Reply context fetch. 2000 char message splitting. Platform wildcard routing: discord: config applies to all Discord JIDs.
whapd — WhatsApp shipped
TypeScript daemon using baileys (Signal protocol). QR auth, session persistence.
JID: whatsapp:<jid>. Offline queue drains on reconnect. Media: photo, video, voice, audio, document, sticker. File upload via baileys. Registers with router HTTP API on startup.
emaid — Email shipped
IMAP TLS polling (every 30s) + SMTP STARTTLS replies. Thread tracking.
JID: email:<threadID>. Local SQLite for thread tracking. Exponential backoff reconnect. Headers injected as context (From, Subject, Date). Config: EMAIL_IMAP_HOST, EMAIL_SMTP_HOST, EMAIL_ACCOUNT, EMAIL_PASSWORD.
mastd — Mastodon shipped
WebSocket streaming for mention notifications. REST API for posts and replies.
JID: mastodon:<...>. Social verbs (boost, edit) carry weight 0 in impulse gate. Config: MASTODON_INSTANCE_URL, MASTODON_ACCESS_TOKEN. Service template: template/services/mastd.toml.
bskyd — Bluesky shipped
Polls AT Protocol notifications every 10s. Session persistence.
JID: bluesky:<...>. Creates posts/replies via XRPC. Config: BLUESKY_IDENTIFIER, BLUESKY_PASSWORD. Service template: template/services/bskyd.toml.
reditd — Reddit shipped
OAuth2 password grant. Polls inbox + configured subreddits every 30s.
JID: reddit:<...>. Replies via /api/comment. Config: REDDIT_CLIENT_ID, REDDIT_CLIENT_SECRET, REDDIT_USERNAME, REDDIT_PASSWORD, REDDIT_SUBREDDITS. Service template: template/services/reditd.toml.

Services & Packages

gated
Gateway daemon. Main loop, routing, containers, impulse gate.
Wires core + store + gateway + api + chanreg + ipc + auth. Polls for new messages every 2s, applies impulse gate, dispatches to queue. Handles /new /ping /chatid /stop. Per-chat error tracking with cursor rollback.
timed
Scheduler daemon. Polls scheduled_tasks, inserts messages into shared DB.
Reads DATABASE env. Polls due tasks every 60s. Cron via robfig/cron. Inserts prompt as message (sender: scheduler) so gateway picks it up in its normal poll loop. Own migration runner. One-shot tasks: next_run set to NULL after firing.
dashd
Operator dashboards. HTMX pages, read-only SQLite.
Pages: portal, status, tasks, activity, groups, memory browser. 10s auto-refresh on tasks/activity. Activity table shows Time/Source/Chat/Sender/Group/Verb/Content. Memory browser: MEMORY.md, CLAUDE.md, diary per group. Auth via JWT session cookie.
proxyd
Web proxy. Auth gate for /dash/, OAuth callbacks, WebDAV, Vite.
All /dash/ routes require valid JWT. OAuth: Google, GitHub, Discord, Telegram Login Widget. /dav/ reverse proxy to dufs container when WEBDAV_ENABLED=true. Slink token resolution (10 req/min per IP). Vite dev server proxy.
onbod
Onboarding daemon. State machine for new user registration.
Auto-included in compose when ONBOARDING_ENABLED=true. State machine: awaiting_name → pending → approved/rejected. Operator /approve creates group + routes + welcome message, seeds default compact-memories cron tasks. Notifies tier-0 JIDs. Filtered by ONBOARDING_PLATFORMS env.
ipc
MCP server library. One unix socket per group per container run.
mark3labs/mcp-go. Socket at ipc/<folder>/gated.sock. Tools registered from MCP manifest, filtered by grants rules for the caller's group. set_grants/get_grants tools allow agents to read/write their own grant rules. delegate_group calls NarrowRules before persisting.
grants
Grant rule engine. Controls MCP tool access and share mount access.
Rule format: [!]action[(param=glob,...)]. Last-match-wins. CheckAction, NarrowRules, MatchingRules, DeriveRules. Default rules derived by tier at container spawn; injected into start.json. MCP manifest filtered so agents only see permitted tools.
chanlib
Shared library for all Go channel adapters.
RouterClient (HTTP client for POST /v1/messages), InboundMsg (with verb field), auth middleware. All Go adapters (teled, discd, mastd, bskyd, reditd, emaid) import chanlib. Reduces per-adapter boilerplate to platform-specific polling and message formatting.
auth
Identity, authorization, JWT, OAuth, middleware.
JWT with 30-day rotating refresh tokens. OAuth: Google (workspace hint via hd=), GitHub (GITHUB_ALLOWED_ORG gate), Discord. Telegram Login Widget with replay protection. Login rate limiting: 5 attempts/15min per IP. Secure cookie flag from HTTPS URL detection.
compose
Docker-compose generation from *.toml service configs.
Services loaded from template/services/*.toml + instance services/*.toml. Container naming: arizuko_<daemon>_<flavor>. onbod auto-included when ONBOARDING_ENABLED=true. davd included when WEBDAV_ENABLED=true. arizuko generate <instance> writes the file without starting anything.
store
SQLite persistence. 15+ tables, WAL mode, 5s busy timeout.
Tables: chats, messages (with verb), registered_groups, routes (with impulse_config JSON), sessions (composite PK: folder+topic), session_log, system_messages, scheduled_tasks, router_state, auth_users, auth_sessions, user_groups, email_threads, onboarding. Migration via PRAGMA user_version.
container
Docker spawn, 8 volume mount types, sidecars, skills seeding.
Builds mount set (group, media, self, share, session, ipc, web, extra). Seeds settings.json with env vars and MCP server config (socat bridge). Seeds skills on first run. Manages sidecar container lifecycle (start, settings wiring, stop). Container name: arizuko-<folder>-<ts> or arizuko-<folder>-task-<id>.

Also: router/ (XML formatting, 7-tier routing rules), queue/ (per-group concurrency, circuit breaker), gateway/ (message loop, impulse gate), diary/ (YAML frontmatter annotations), chanreg/ (channel registry, health loop), api/ (HTTP handlers), notify/ (operator fan-out), mountsec/ (mount allowlist), groupfolder/ (path resolution), core/ (types, Channel interface), cmd/arizuko/ (CLI).

Build & Run

$ make build && make images && make agent 
# create an instance
arizuko create foo

# edit config (.env in data dir)
vim /srv/data/arizuko_foo/.env

# register a group
arizuko group foo add tg:-123456789

# generate compose and start
arizuko generate foo
docker compose -f /srv/data/arizuko_foo/docker-compose.yml up

SQLite for state. Docker for agent isolation and compose orchestration. No external dependencies beyond docker. Systemd units call arizuko generate (ExecStartPre) then docker compose up (ExecStart) — no host binary needed at runtime.

Roadmap

MCP tooling
1
get_history MCP tool
Expose message history to agents via MCP. Required by the recall-messages skill. Agent can query past messages from a group without reading raw files.
2
Share mount controlled by grants
share_mount grant rule controls whether a group's container gets the /workspace/share mount. Currently controlled by tier only. Grants rule would allow fine-grained per-group opt-in/out.
Storage
1
WebDAV per-group ACL
Currently /dav/ mounts all groups/ read-only for any authenticated user. Identity-based redirect: JWT groups claim → restrict to specific group subfolder.
Channels
1
X/Twitter adapter
OAuth 2.0, rate-limited posting. Middleware intercepts post tool calls into a pending queue. No draft publishes without human approval.
Features crystallize from production usage. Specs in specs/7/ track in-progress designs.

Agent context
